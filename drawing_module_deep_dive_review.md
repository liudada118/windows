# 画图模块需求深度评估报告

**评估结论：** 否，尚未达到足够深度。

**核心问题：** 当前文档（PRD V5.1）在“画什么”（What）和“为什么这样画”（Why）方面做得非常出色，但在“**如何精确地画**”（How）的微观交互和“**如何应对异常**”（Exceptions）方面，存在大量细节缺失。开发人员无法仅凭此文档处理所有交互细节和边界情况。

以下是详细的缺失项分析：

---

## 1. 数据模型与 Schema (第4章)

**现状：** 定义了所有核心数据结构，并给出了完整示例。

**缺失细节：**

| 缺失项 | 具体描述 | 对开发的影响 |
| :--- | :--- | :--- |
| **字段约束未定义** | `sash.openDirection` 和 `sash.hingeSide` 的联动关系是什么？（例如，左开扇的铰链侧必须是左侧）。`mullion.position` 是相对坐标还是绝对坐标？ | 开发人员需要猜测或重新沟通这些核心业务规则，容易出错。 |
| **ID 生成策略缺失** | `frame_001`, `mullion_001` 这些 ID 是前端生成还是后端生成？生成规则是什么？（UUID? 自增？） | 前后端联调时会产生 ID 冲突或不一致的问题。 |
| **枚举值不完整** | `sash.type` 只列了6种，但模板库里有30+种扇，这些扇的 `type` 值是什么？`frame.type` 也是同样问题。 | 开发人员无法穷举所有扇类型，导致渲染和算料逻辑覆盖不全。 |
| **组合窗 `windowRef` 机制模糊** | `windowRef` 指向的 `design_id` 是哪个表的 ID？`window_design` 还是 `window_template`？子窗和主窗的 `metadata` 如何合并？ | 组合窗功能无法实现，因为最关键的数据关联机制不明确。 |

---

## 2. 画布交互与渲染 (第7章)

**现状：** 定义了工具箱、属性面板和“黄金标准”UX原则。

**缺失细节：**

| 缺失项 | 具体描述 | 对开发的影响 |
| :--- | :--- | :--- |
| **“三位一体”交互未落地** | PRD 定义了模板驱动、自由拖拽、精确输入三种模式，但没有说明它们如何**具体结合**。例如，拖拽中梃时，尺寸标注是实时变化还是拖拽结束后再变化？拖拽时是否可以临时切换到精确输入？ | 开发人员只能实现孤立的三种模式，无法做出流畅的、三位一体的交互体验。 |
| **智能吸附规则缺失** | “智能吸附”只是一个概念。吸附的阈值是多少像素？优先吸附到网格还是对齐线？吸附到等分点时的视觉提示是什么？ | “智能吸附”功能无法实现，或者实现效果与产品设想大相径庭。 |
| **控制点交互未定义** | 选中组件后出现的“控制手柄”是什么样的？矩形有几个控制点（4个角+4条边）？拖动不同的控制点分别对应什么操作（等比缩放？自由变形？）？ | 组件的编辑功能无法实现，因为最基本的控制点交互没有定义。 |
| **渲染细节不足** | “45度斜切角”如何绘制？是用图片还是 Canvas Path？“3D立体效果”的光源方向是哪里？阴影如何计算？“木纹贴图”的UV映射规则是什么？ | 渲染效果完全依赖开发人员的个人理解，无法保证视觉效果的统一性和准确性。 |
| **错误与边界状态未定义** | 拖拽中梃导致某个分格小于最小尺寸时，具体如何提示？是红色边框闪烁，还是弹出文字提示？加载一个损坏的 `design_data_json` 时，画布应该如何展示？ | 缺少这些定义，程序在遇到异常情况时容易崩溃，或者用户体验非常糟糕。 |
| **性能指标缺失** | “分层渲染”、“脏矩形重绘”都是很好的策略，但没有定义具体的性能目标。例如，在包含200个组件的画布上，拖拽操作的帧率（FPS）应不低于多少？ | 开发人员没有明确的优化目标，可能导致在复杂场景下应用卡顿。 |

## 3. 操作流程与状态机 (第7章)

**现状：** 每个工具的交互流程描述为"点击模板 → 点击画布 → 放置组件"的线性流程。

**缺失细节：**

| 缺失项 | 具体描述 | 对开发的影响 |
| :--- | :--- | :--- |
| **操作取消/中断流程** | 用户选择了框模板后，如何取消放置？按 Esc？点击空白区域？右键？如果在放置过程中切换到其他工具Tab，当前操作如何处理？ | 用户无法优雅地取消操作，体验割裂。 |
| **多选与批量操作** | 能否框选多个组件？能否 Ctrl+Click 多选？多选后能否批量删除、批量修改属性？ | 复杂设计中，用户需要逐个操作每个组件，效率极低。 |
| **拖拽中梃的完整状态机** | 中梃拖拽涉及：鼠标按下 → 开始拖拽 → 实时预览新位置 → 约束检查 → 释放 → 更新所有关联区域/扇/玻璃的尺寸。这个完整的状态转换流程没有定义。 | 中梃拖拽是最核心的交互之一，缺少状态机定义会导致实现混乱。 |
| **撤销/恢复的粒度** | "操作栈最大50步"，但什么算"一步"？拖拽中梃从位置A到位置B是一步，还是每移动1px都是一步？修改右侧面板的多个字段是一步还是多步？ | 撤销/恢复行为不可预测，用户体验差。 |
| **复制/粘贴规则** | 复制一个扇时，是否连同其中的玻璃、格条一起复制？粘贴时放在哪个区域？如果目标区域尺寸不同，扇如何适配？ | 复制粘贴功能无法正确实现。 |

---

## 4. 导出与打印 (第7章)

**现状：** 仅提到"下载截图 → 导出为PNG"和"打印 → Ctrl+P"。

**缺失细节：**

| 缺失项 | 具体描述 | 对开发的影响 |
| :--- | :--- | :--- |
| **导出格式** | 除了 PNG，是否支持 PDF、DXF（CAD格式）、SVG？每种格式的导出规格是什么？ | 工厂通常需要 DXF 格式导入到 CNC 设备，仅支持 PNG 无法满足生产需求。 |
| **打印模板** | 打印时的纸张大小？是否有固定的打印模板（如包含公司Logo、客户信息、图纸、算料清单的综合打印页）？ | 打印出来的图纸不专业，缺少必要的业务信息。 |
| **截图分辨率** | 导出 PNG 的分辨率是多少？是否支持 2x/3x 高清导出？背景是否透明？ | 导出的图片模糊或不符合客户要求。 |
| **批量导出** | 一个订单有多个窗型，能否一键导出所有窗型的图纸？ | 用户需要逐个窗型手动导出，效率低下。 |

---

## 5. 3D 渲染 (第7.7章)

**现状：** 定义了旋转、缩放、颜色切换和爆炸视图。

**缺失细节：**

| 缺失项 | 具体描述 | 对开发的影响 |
| :--- | :--- | :--- |
| **2D → 3D 映射规则** | 2D 的 `design_data_json` 如何转换为 3D 模型？型材的截面形状如何拉伸为3D？玻璃的厚度如何表现？ | 3D 渲染功能无法实现，因为最核心的数据映射规则缺失。 |
| **3D 模型精度** | 3D 模型是"示意图级别"还是"工程级别"？是否需要展示型材的内部腔体结构？ | 开发人员不知道需要做到什么精度，可能过度开发或不足。 |
| **光照与材质** | 光源类型（环境光+方向光？）、强度、位置？金属材质的反射率？玻璃的折射率？ | 3D 渲染效果完全取决于开发人员的审美，无法保证质量。 |
| **开启动画** | 是否支持模拟门窗的开启/关闭动画？如果支持，动画的时长、缓动曲线是什么？ | 这是一个非常有吸引力的功能，但缺少定义就无法实现。 |

## 6. 综合评估

### 6.1 已做好的部分（优势）

| 维度 | 评分 | 说明 |
| :--- | :--- | :--- |
| 数据结构定义 | ★★★★☆ | Schema 完整，有完整 JSON 示例，字段定义清晰 |
| 工具箱分类 | ★★★★★ | 8个Tab、200+种模板，分类详尽 |
| 页面布局 | ★★★★☆ | ASCII 线框图清晰，面板结构明确 |
| 渲染规则 | ★★★☆☆ | 有基础规则，但缺少精确的绘制参数 |
| UX 原则 | ★★★★☆ | "黄金标准"章节提供了优秀的方向性指导 |

### 6.2 仍然缺失的部分（短板）

| 维度 | 评分 | 说明 |
| :--- | :--- | :--- |
| 微观交互定义 | ★★☆☆☆ | 缺少控制点、吸附、拖拽状态机等核心交互细节 |
| 边界/异常处理 | ★☆☆☆☆ | 几乎没有定义任何异常场景的处理方式 |
| 导出/打印规格 | ★☆☆☆☆ | 仅提到 PNG 截图和 Ctrl+P，远远不够 |
| 3D 映射规则 | ★☆☆☆☆ | 2D→3D 的转换逻辑完全缺失 |
| 性能指标 | ★★☆☆☆ | 有优化策略但无量化目标 |
| 字段联动规则 | ★★☆☆☆ | Schema 字段之间的业务约束关系未定义 |

### 6.3 总体评分

**当前深度：60/100 分**

画图模块的"骨架"已经非常完整（数据结构 + 工具分类 + 页面布局），但"血肉"（微观交互 + 异常处理 + 精确渲染参数）严重不足。

如果将这份文档交给一个**资深前端开发**，他可以搭建出画图模块的基本框架，但在实现每一个具体交互时，都需要频繁地回来找产品经理确认细节。

### 6.4 建议的补充优先级

| 优先级 | 补充内容 | 理由 |
| :--- | :--- | :--- |
| **P0** | 中梃拖拽完整状态机 + 控制点交互定义 | 这是画图模块最核心的两个交互，缺了它们画图功能形同虚设 |
| **P0** | Schema 字段联动规则（openDirection↔hingeSide、position的坐标系） | 没有这些规则，数据就是"死"的 |
| **P0** | 2D→3D 映射规则 | 没有映射规则，3D渲染无法启动开发 |
| **P1** | 导出/打印完整规格（PDF模板、DXF格式、批量导出） | 生产环节的刚需 |
| **P1** | 所有异常/边界场景的处理方式 | 决定产品的健壮性 |
| **P1** | 撤销/恢复粒度定义 + 多选/批量操作规则 | 决定编辑效率 |
| **P2** | 渲染精确参数（光源、材质、阴影） | 决定视觉品质 |
| **P2** | 性能量化指标（FPS、加载时间） | 决定流畅度 |
