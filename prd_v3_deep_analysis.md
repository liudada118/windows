# 《PRD_V3.md》深度审查与分析报告

**版本：1.0**

**作者：Manus AI**

**日期：2026-03-02**

---

## 1. 引言

本报告旨在对 `liudada118/windows` 项目的核心产品需求文档（PRD_V3.md）进行一次全面、深入的审查。首先必须承认，`PRD_V3.md` 是一份在广度和深度上都表现出色的文档，其对门窗设计、订单管理、车间生产等核心模块的描述详尽，体现了产品团队对业务的深刻理解。然而，从更严苛的产品管理和技术实现视角审视，该文档在**战略完备性、业务逻辑闭环、技术架构严谨性**以及**用户体验关怀**等多个维度，仍存在显著的短板和盲区。这些问题若不加以解决，将可能在开发过程中引发重大返工，甚至导致产品在市场竞争中处于不利地位。

本报告将系统性地揭示这些短板，并参照行业最佳实践提出改进建议，旨在帮助团队打造一款真正健壮、可扩展且具备核心竞争力的 SaaS 产品。

## 2. 核心问题：战略与业务逻辑盲区

此部分揭示了 PRD 中最严重的问题，它们直接关系到产品的核心价值主张和商业模式的成败。

### 2.1 算料与下料优化定义缺失：核心引擎的“空心化”

门窗 SaaS 软件的核心竞争力在于其**算料的准确性**和**下料的优化率**。这直接决定了工厂的成本控制和生产效率。然而，`PRD_V3.md` 在这两个最关键的环节上却出现了致命的定义缺失。

- **算料公式引擎未定义**：文档在 8.2 节提到了10种公式 Tab 和 `formula_json` 字段，但完全没有定义公式的**语法规范**。公式是简单的数学表达式，还是需要支持条件判断、循环、函数调用的脚本语言？它如何处理不同型材系列的复杂搭接、扣减、损耗规则？这一定义的缺失，使得整个算料引擎成为了一个无法估算开发难度、无法保证计算准确性的“黑盒”。

- **下料优化算法完全空白**：`CHANGELOG.md` 将“型材优化下料”列为 P0 级（最高优先级）功能，但 `PRD_V3.md` 全文对此只字未提。一维型材切割优化（1D Cutting Stock Problem）是复杂的算法问题，需要明确**优化目标**（例如，是追求最高材料利用率，还是最少切割次数？）、**约束条件**（例如，切割机刀口宽度、原材料长度规格、余料回收策略等）。缺少这些定义，开发团队将无从下手，也无法保证软件能真正为工厂节省成本。

> **行业对标**：极致下料、仁霸等竞品均将“智能算法”、“优化排版”作为核心卖点。国际领先的解决方案如 RA Workshop 和 Soft Tech V6 更是提供了与 CNC 切割设备直接对接的自动化能力。当前 PRD 在此领域的空白，是战略性的短板。

### 2.2 核心业务流程中断：未形成完整闭环

软件定义了一个从“意向单”到“已发货”的流程，但在多个关键节点戛然而止，未能覆盖门窗业务的全生命周期，也缺少对真实世界复杂性的应对。

| 缺失环节 | 业务影响 | 建议补充 |
| :--- | :--- | :--- |
| **订单变更/补单** | 客户在生产过程中修改尺寸、增补门窗是常态。PRD 缺乏变更流程定义，将导致数据混乱和生产错误。 | 定义订单锁定机制、变更审批流程、差价计算规则、以及与生产和财务模块的联动。 |
| **安装与售后** | 门窗业务在“发货”后远未结束，安装交付和售后服务是关键环节。PRD 的缺失导致软件无法管理安装排期、安装人员、验收状态和售后维修记录。 | 增加“安装管理”和“售后服务”模块，覆盖安装排班、进度跟踪、完工验收、客户报修、维修派单等功能。 |
| **退款与退货** | 财务模块只有收款和支出，完全没有定义退款和退货流程。这在真实商业场景中是不可或缺的。 | 在财务模块补充退款单、退货入库、以及与订单和库存状态联动的完整流程。 |

### 2.3 关键模块定义空洞：名为“AI”，实为空白

PRD 中罗列了多个富有吸引力的高阶功能，如 AI 炫图、阳光房 3D 设计、AR 拍照搭配等。然而，这些模块的定义极其粗略，几乎为空白，这构成了巨大的“需求陷阱”。

- **模块十（炫图/AI渲染）**：仅有5行文字描述。作为一个需要调用第三方 AI 服务的复杂功能，PRD 必须定义：AI 服务商选型、API 接口协议、成本控制与用量限制、效果评估标准、渲染失败处理、异步任务管理等。
- **模块十一（阳光房3D设计）**：仅列出6个功能名称。这本质上是一个独立的、比门窗设计更复杂的 3D 建模工具。PRD 需要从零开始定义其数据模型、交互逻辑、渲染引擎技术选型、以及与主流程的集成方案。
- **模块十二（拍照搭配/AR）**：同样只有简单的流程描述。AR 功能涉及复杂的技术选型（WebXR, ARKit/ARCore）、摄像头权限管理、三维空间识别与追踪、光照模拟、模型渲染等，这些在 PRD 中均未体现。

这种“占位式”的需求定义是极其危险的，它给予了团队一个功能已经设计完成的假象，但实际上隐藏了巨大的技术不确定性和无法估量的工作量。

## 3. 关键短板：数据与技术架构漏洞

如果说战略盲区决定了产品“能不能成”，那么技术架构的漏洞则决定了产品“能不能活”。

### 3.1 数据隔离方案存在严重漏洞

这是 `PRD_V3.md` 中最紧急、最严重的技术漏洞。作为一个多租户 SaaS 系统，数据隔离是其生命线。

- **`tenant_id` 字段大面积缺失**：非功能性需求（19.2节）明确要求使用 `tenant_id` 进行数据隔离，但 PRD 中定义的 **22 张核心数据表中，没有任何一张包含了 `tenant_id` 字段**。这是一个灾难性的疏忽。
- **隔离逻辑混乱**：部分表（如 `user`）似乎试图使用 `owner_user_id` 进行隔离，但大量核心业务表（如 `order`, `payment`, `expense`, `production_task` 等）既没有 `tenant_id` 也没有 `owner_user_id`。这意味着，一个租户的订单、财务数据将可能被其他租户轻易访问，存在致命的数据安全风险。

> **修复建议**：立即暂停所有功能开发，由架构师牵头，重新审查并修正所有数据表结构，确保 `tenant_id` 作为强制性字段存在于每一个需要隔离的表中，并建立严格的后端数据访问策略。

### 3.2 数据模型不一致与不完整

除了数据隔离问题，数据模型自身也充满了矛盾和缺失。

| 问题点 | 描述 | 影响 |
| :--- | :--- | :--- |
| **主键类型冲突** | API 响应中 `user.id` 和 `customerId` 均为整数（如 1001），但数据模型中对应的 `user_id` 和 `customer_id` 均定义为 `VARCHAR(36)` 的 UUID。 | 前后端联调时将出现严重的数据类型不匹配错误。 |
| **状态定义冲突** | `order` 表的 `status`（意向单/合同单）和 `production_status`（生产状态）职责不清，车间管理模块的状态机逻辑混淆了两者。 | 导致订单状态无法准确追踪，业务逻辑混乱。 |
| **缺少审计日志表** | 安全需求中提到的“关键操作记录审计日志”没有对应的数据表 `audit_log`。 | 系统缺乏可追溯性，出现问题难以排查和定责。 |
| **缺少软删除字段** | 多处提到“软删除”，但相关数据表（如 `order`）普遍缺少 `deleted_at` 或 `is_deleted` 字段。 | 无法实现真正的软删除功能，只能进行物理删除，有数据丢失风险。 |

### 3.3 API 设计粗糙且缺失

API 是前后端协作的契约，其质量直接影响开发效率。当前 PRD 的 API 设计存在大量问题。

- **大量“骨架”API**：许多 API（如 `PUT /api/orders/{id}`）只有一个标题，没有定义详细的请求体、响应体和错误码，这使得前端无法开展工作。
- **缺少批量操作 API**：UI 层面定义了“批量转正式单”等功能，但后端没有提供对应的批量处理接口，这将导致前端需要通过循环调用单个 API 来模拟，效率低下且容易出错。
- **缺少文件上传 API**：系统多处需要上传图片（Logo、木纹贴图、AR场景图等），但 PRD 未定义统一的文件上传接口和管理方案。
- **缺少实时通信机制**：在多角色协作场景下（如订单状态变更、车间任务分配），系统应通过 WebSocket 等技术实现实时消息推送，但相关 API 和设计完全缺失。

## 4. 潜在风险：用户体验与非功能性需求

这部分问题虽然不如前两者致命，但直接影响产品的品质和用户的长期留存。

### 4.1 用户体验设计缺失

PRD 过度关注“功能实现”（Happy Path），而严重忽略了真实的用户使用场景和体验细节。

- **异常状态未定义**：网络中断、保存失败、并发编辑冲突、权限不足等异常场景如何处理？PRD 中完全没有涉及。
- **过渡状态未定义**：页面加载（Loading）、列表为空（Empty State）、操作等待等过渡状态缺乏设计，将导致用户体验生硬、充满不确定性。
- **缺少设计约束**：画图模块没有定义业务约束规则，例如中梃的最小间距、平开扇的最大尺寸等。这可能导致用户设计出无法生产的窗型。
- **缺少新手引导（Onboarding）**：对于一个复杂的专业 SaaS 工具，PRD 没有规划任何形式的新手引导、功能介绍或帮助文档体系，新用户的学习曲线将非常陡峭。

### 4.2 非功能性需求考虑不周

非功能性需求定义了系统的健壮性、可维护性和扩展性，PRD 在此方面同样存在不足。

- **指标缺乏依据**：“支持 500 并发用户”的目标是如何得出的？缺少业务预期的支撑，这个数字显得很随意。
- **缺少关键SLA**：没有定义系统的**可用性目标**（如 99.9%），这对于 SaaS 服务是至关重要的商业承诺。
- **国际化考虑不足**：虽然 `CHANGELOG.md` 提及多语言支持，但数据模型和 API 设计均未考虑国际化（i18n）需求，例如多语言字段、时区处理等。
- **运维需求缺失**：没有定义**监控、告警、日志**等关键的运维（DevOps）需求，这将导致系统上线后成为一个难以维护的“黑盒”。

## 5. 总结与改进建议

`PRD_V3.md` 是一份内容丰富但充满“陷阱”的需求文档。它成功地描绘了产品的功能蓝图，却在决定产品成败的核心战略、技术架构和业务闭环上留下了巨大的空白和风险。如果不经修订直接进入开发，可以预见项目将陷入无尽的需求澄清、技术争论和频繁返工之中。

为此，我提出以下分级改进建议：

| 优先级 | 改进项 | 核心任务 |
| :--- | :--- | :--- |
| **P0 (最高)** | **修复架构基石** | 1. **数据隔离**：立即修复 `tenant_id` 缺失的致命漏洞。<br>2. **核心算法**：成立专项小组，与行业专家合作，完成**算料公式引擎**和**下料优化算法**的详细定义。 |
| **P1 (高)** | **完善业务闭环** | 1. **补全流程**：定义订单变更、安装售后、退款退货等缺失的业务流程。<br>2. **填充模块**：为 AI 炫图、阳光房、AR 搭配等关键模块撰写独立的、详细的 PRD。 |
| **P2 (中)** | **提升产品品质** | 1. **完善 API**：补全所有 API 的详细设计，增加批量操作和实时通知接口。<br>2. **补充 UX/NFR**：定义异常/过渡状态，补充可用性、国际化、可运维性等非功能性需求。 |

建议产品、技术、业务团队立即组织一次针对本报告的专题评审会，逐项对齐认知，并基于以上建议重新制定一份切实可行、逻辑严密、技术可靠的 `PRD_V4.0`。只有地基稳固，高楼才能屹立不倒。
