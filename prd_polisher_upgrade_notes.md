# prd-polisher 技能升级点

从《门窗之星》产品文档体系分析报告（V2）中，可以提炼出以下可复用的审查框架和方法论，用于升级 `prd-polisher` 技能。

## 1. 五维评分模型

这是报告的核心亮点，提供了一个结构化的、可量化的 PRD 评估框架。应将其作为技能中“深度审查”阶段的核心方法。

| 维度 | 审查要点 |
| :--- | :--- |
| **文档治理与组织** | 单一事实来源、版本历史、归档策略、README 清晰度 |
| **需求完整性** | 功能模块覆盖率、核心业务流程闭环、空洞模块（只有标题没有内容）识别 |
| **技术可执行性** | 关键算法/DSL 是否有伪代码或 EBNF、技术选型是否锁定、性能指标是否量化 |
| **内部一致性** | 跨文档（PRD vs ARCHITECTURE）和文档内部（章节之间）是否存在矛盾 |
| **产品商业性** | 是否包含竞品分析、用户画像、北极星指标、MVP 边界定义 |

## 2. 逐章审计清单

报告的第 5 节提供了一个非常好的逐章审计模板，可以将其泛化，作为技能的一部分。

- **产品概述**：检查产品定位、核心价值、竞品分析、用户画像、北极星指标。
- **数据模型**：检查表结构完整性、索引设计、数据量估算、DDL 脚本。
- **核心模块（如画图）**：检查页面布局、工具箱定义、交互流程、微观交互、模板数据、撤销/恢复粒度。
- **技术模块（如算料引擎）**：检查 DSL 语法定义（EBNF）、错误处理、安全策略。
- **API 接口**：检查全局约定、接口覆盖率、请求/响应示例均匀性、OpenAPI 规格、限流策略。
- **非功能性需求**：检查性能、并发、可用性、安全、监控、兼容性、国际化、压测方案。

## 3. 行动计划模板

报告的第 6 节提供了一个清晰的、可落地的行动计划模板，包含优先级、负责人、工时和产出。这应该成为技能交付审查报告时的标准格式。

- **立即行动（本周内）**：主要用于修复明显的、低成本的文档内部矛盾。
- **短期行动（1-2 周内）**：主要用于定义 MVP 范围、关键技术选型 PoC、补充核心技术规格。
- **中期行动（2-4 周内）-**：主要用于撰写产品战略、补齐次要模块的详细规格。
- **长期行动（V-Next）**：主要用于非 MVP 范围内的模块的技术预研。

## 4. 缺失清单补齐率审计

如果审查是基于一份已有的“缺失清单”进行的，那么最终的报告应该包含一个“补齐率审计”章节，用表格清晰地展示每项缺失的补齐状态（完全补齐/部分补齐/未补齐），并计算最终的补齐率。这使得改进的效果一目了然。

## 技能工作流升级

基于以上提炼，`prd-polisher` 的工作流可以升级为：

1.  **阶段一：文档治理审查**（原阶段一）
    -   盘点、瘦身、建立索引。
    -   **新增**：使用“文档治理与组织”维度对现状进行评分。
2.  **阶段二：产品战略与技术可行性审查**（全新阶段）
    -   使用“产品商业性”和“技术可执行性”维度进行审查。
    -   检查竞品分析、用户画像、北极星指标、MVP 边界。
    -   检查关键算法、技术选型、性能指标。
3.  **阶段三：需求完整性与一致性审查**（原阶段二）
    -   使用“需求完整性”和“内部一致性”维度进行审查。
    -   逐章审计，找出空洞模块和跨文档矛盾。
4.  **阶段四：输出审查报告与行动计划**（全新阶段）
    -   生成五维评分雷达图。
    -   输出包含“逐章审计”和“核心短板分析”的报告。
    -   提供“立即/短期/中期/长期”的行动计划表。
5.  **阶段五：深度补齐与整合**（原阶段三）
    -   根据行动计划，执行补齐和整合工作。
6.  **阶段六：最终交付与说明**（原阶段四）
    -   更新或创建 PRD_README.md。
