## 第一部分：缺失项补齐的详细分工计划

> 本部分将 36 项缺失按 P0→P1→P2 优先级拆解为可执行的工作任务，明确每项的负责人、工时、交付物和依赖关系。缺失项补齐的本质是**文档补写 + 技术方案设计 + 数据模型修正**，不涉及编码实现。

### 一、P0 级缺失补齐（第 1 周，阻塞性问题）

P0 级共 8 项，分为三个工作包并行推进。如果这 8 项不完成，后续所有开发工作都不应启动，因为数据隔离方案决定了整个后端的 SQL 查询模式，算料公式引擎决定了前端核心模块的技术架构，下料优化算法决定了是否需要引入额外的算法服务。

### 六、风险管理与应对策略

| 风险 | 概率 | 影响 | 应对策略 |
| :--- | :--- | :--- | :--- |
| 算料公式引擎复杂度超预期 | 高 | 延期 2-3 周 | Sprint 4 预留 buffer，TL 提前介入 PoC 验证 |
| 下料优化算法性能不达标 | 中 | 影响 V1.1 | 先用 FFD 贪心算法保底，后续迭代优化 |
| 后端并发性能不足 | 中 | 影响上线 | 提前做压测，预留水平扩展方案（Docker + 负载均衡） |
| AI 渲染服务成本过高 | 中 | 影响 V2.0 ROI | 设置用量上限，提供付费升级选项 |
| 前端 A 成为瓶颈（画布 + 算料 + 3D 全在一人） | 高 | 全局延期 | TL 分担算料引擎部分工作，必要时招聘第三位前端 |
| 门窗行业规则理解偏差 | 高 | 返工 | 每个 Sprint 结束邀请门窗厂用户试用反馈 |
