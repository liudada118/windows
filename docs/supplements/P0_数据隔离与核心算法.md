## S1. 多租户数据隔离方案

### S1.1 隔离策略

本系统采用**共享数据库 + 行级隔离**（Row-Level Security）方案。所有需要隔离的数据表必须包含 `tenant_id` 字段，后端在每次数据库查询时强制附加 `WHERE tenant_id = :current_tenant_id` 条件。

**租户定义**：一个租户对应一个门窗企业（工厂或门店），由管理员账号注册时自动创建。管理员账号下的所有子账号共享同一 `tenant_id`。

### S1.2 `tenant` — 租户表（新增）

| 字段名 | 数据类型 | 键 | 可空 | 默认值 | 描述 |
|---|---|---|---|---|---|
| tenant_id | VARCHAR(36) | PK | N | UUID | 租户唯一标识 |
| name | VARCHAR(100) | — | N | — | 企业名称 |
| admin_user_id | VARCHAR(36) | FK→user | N | — | 管理员账号 |
| plan | ENUM('试用','门店版','工厂版') | — | N | '试用' | 订阅计划 |
| trial_expires_at | DATE | — | Y | NULL | 试用到期日 |
| max_users | INT | — | N | 5 | 最大子账号数 |
| storage_quota_mb | INT | — | N | 1024 | 存储配额 MB |
| created_at | DATETIME | — | N | NOW() | 创建时间 |

### S1.4 后端强制隔离规则

1. **中间件层**：所有 API 请求经过 `TenantMiddleware`，从 JWT Token 中提取 `tenant_id`，注入到请求上下文。
2. **ORM 层**：使用全局查询过滤器（如 Prisma 的 `$allOperations` 或 TypeORM 的 `Subscriber`），自动为所有 SELECT/UPDATE/DELETE 语句附加 `tenant_id` 条件。
3. **INSERT 层**：所有新增记录自动填充当前 `tenant_id`，禁止前端传入。
4. **管理后台**：超级管理员（平台运营方）可跨租户查询，使用独立的管理 API 路径 `/admin/api/`。

## S2. 算料公式引擎规格书

### S2.1 概述

算料公式引擎是系统的核心计算模块，负责根据门窗设计数据（`design_data_json`）和型材系列配置的公式（`calculation_rule.formula_json`），自动计算出型材下料尺寸、玻璃尺寸、五金配件数量以及最终报价。

### S2.2 公式语言规范（WindoorFormula DSL）

采用自定义领域特定语言（DSL），语法接近 Excel 公式，降低门窗行业用户的学习成本。

#### S2.2.1 数据类型

| 类型 | 示例 | 说明 |
|---|---|---|
| Number | `2400`, `3.14`, `-10` | 浮点数，精度到小数点后 4 位 |
| String | `"中空玻璃"`, `"5+12A+5"` | 字符串，双引号包裹 |
| Boolean | `TRUE`, `FALSE` | 布尔值 |
| Array | `[1, 2, 3]` | 数组（仅用于函数参数） |

#### S2.2.2 运算符

| 运算符 | 含义 | 优先级 | 示例 |
|---|---|---|---|
| `()` | 括号 | 最高 | `(W - 2 * FW)` |
| `*`, `/`, `%` | 乘、除、取模 | 高 | `W * H / 1000000` |
| `+`, `-` | 加、减 | 中 | `W - 2 * FW + 10` |
| `>`, `<`, `>=`, `<=`, `==`, `!=` | 比较 | 低 | `W > 1500` |
| `&&`, `\|\|`, `!` | 逻辑与、或、非 | 最低 | `W > 1500 && H > 1200` |

#### S2.2.3 内置函数

| 函数 | 参数 | 返回值 | 说明 |
|---|---|---|---|
| `ROUND(value, digits)` | 数值, 小数位数 | Number | 四舍五入 |
| `CEIL(value)` | 数值 | Number | 向上取整 |
| `FLOOR(value)` | 数值 | Number | 向下取整 |
| `MAX(a, b, ...)` | 多个数值 | Number | 取最大值 |
| `MIN(a, b, ...)` | 多个数值 | Number | 取最小值 |
| `ABS(value)` | 数值 | Number | 绝对值 |
| `IF(condition, trueVal, falseVal)` | 条件, 真值, 假值 | Any | 条件判断 |
| `SUM(array)` | 数组 | Number | 求和 |
| `COUNT(array)` | 数组 | Number | 计数 |
| `SQRT(value)` | 数值 | Number | 平方根 |
| `PI()` | 无 | Number | 圆周率 3.14159265 |
| `CONCAT(a, b, ...)` | 多个字符串 | String | 字符串拼接 |
| `LOOKUP(key, table)` | 键, 查找表 | Any | 在查找表中按键取值 |

### S2.3 `formula_json` 结构定义

```json
{
  "version": "1.0",
  "items": [
    {

      "materialCode": "001",
      "materialName": "框上横",
      "category": "型材",
      "formula": "W - 2 * FW + 10",
      "quantityFormula": "2",
      "unit": "mm",
      "condition": "",
      "remarks": "上下横框料"
    },
    {
      "id": "f002",
      "materialCode": "002",
      "materialName": "框竖边",
      "category": "型材",
      "formula": "H - 2 * FW + 10",
      "quantityFormula": "2",
      "unit": "mm",
      "condition": "",
      "remarks": "左右竖框料"
    },
    {
      "id": "f003",
      "materialCode": "G01",
      "materialName": "中空玻璃",
      "category": "玻璃",
      "formula": "(GW * GH) / 1000000",
      "quantityFormula": "1",
      "unit": "㎡",
      "condition": "G_SPEC == \"5+12A+5\"",
      "remarks": ""
    }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| items[].id | string | Y | 公式项唯一 ID |
| items[].materialCode | string | Y | 关联物料编号 |
| items[].materialName | string | Y | 物料名称（冗余显示用） |
| items[].category | string | Y | 分组：型材/玻璃/配件/五金 |
| items[].formula | string | Y | **下料尺寸/面积/数量的计算公式**（WindoorFormula DSL 表达式） |
| items[].quantityFormula | string | Y | **数量公式**（如固定值 "2" 或表达式 "SASH_COUNT * 2"） |
| items[].unit | string | Y | 计算结果单位：mm/㎡/个/支 |
| items[].condition | string | N | 条件表达式，为空或计算结果为 TRUE 时该项生效 |
| items[].remarks | string | N | 备注 |

### S2.4 公式执行流程

```
1. 前端保存设计 → design_data_json 写入数据库
2. 用户点击"报价"或"测试公式" → 前端发送 POST /api/series/{id}/test-formula
3. 后端加载 design_data_json + calculation_rule.formula_json
4. 公式引擎初始化：
   a. 解析 design_data_json，计算所有系统变量（W, H, FW, GW, GH 等）
   b. 加载自定义变量
   c. 构建变量上下文（Variable Context）
5. 遍历 formula_json.items：
   a. 评估 condition 表达式（为空视为 TRUE）
   b. 若 condition 为 TRUE，评估 formula 表达式得到尺寸/面积
   c. 评估 quantityFormula 得到数量
   d. 查询 material 表获取单价
   e. 计算金额 = 尺寸/面积 × 数量 × 单价
6. 汇总所有项，返回算料报价结果
```

### S2.5 公式调试器交互设计

"测试公式"按钮点击后，弹出调试面板：

| 区域 | 内容 |
|---|---|
| 变量监视区 | 表格展示所有变量的当前值（系统变量 + 自定义变量），实时更新 |
| 公式执行区 | 逐行展示每条公式的执行结果：物料名称、公式表达式、计算过程、最终值 |
| 错误提示区 | 红色高亮显示公式语法错误或运行时错误，精确定位到行号和字符位置 |
| 结果汇总区 | 型材清单表格、玻璃清单表格、配件清单表格、总金额 |

**错误处理规则：**

| 错误类型 | 处理方式 |
|---|---|
| 语法错误 | 标红公式行，提示"公式语法错误：第X个字符处" |
| 未定义变量 | 标红变量名，提示"变量 XXX 未定义" |
| 除零错误 | 该项结果显示为 0，标黄警告"除数为零" |
| 类型错误 | 标红，提示"类型不匹配：期望数值，实际为字符串" |

### S2.6 技术选型建议

公式引擎建议在**后端执行**（Node.js 环境），前端"测试公式"通过 API 调用后端计算。理由如下：

1. **安全性**：公式可能包含企业核心算料参数，不应暴露在前端。
2. **一致性**：报价、算料、下料优化均依赖同一引擎，后端统一执行确保结果一致。
3. **性能**：批量算料（100+ 樘）需要后端并行处理。

推荐使用 **mathjs** 库作为表达式解析和计算的基础，在其上扩展自定义函数和变量注入。

## S3. 下料优化算法需求

### S3.1 问题定义

一维下料优化（1D Cutting Stock Problem）：给定一组**需求料**（不同长度的型材切割需求）和**原材料**（标准长度的型材棒料），求出最优的切割方案，使得**材料利用率最高**（即余料最少）。

### S3.2 输入参数

| 参数 | 类型 | 来源 | 说明 |
|---|---|---|---|
| 需求料列表 | Array | 算料引擎输出 | 每项包含：物料编号、名称、下料长度(mm)、数量 |
| 原材料规格 | Array | `material.bar_length` | 每种物料的标准棒料长度（如 6000mm、6500mm） |
| 刀口宽度 | Number | 系统设置 | 切割锯片宽度，默认 5mm |
| 余料回收阈值 | Number | 系统设置 | 大于此长度的余料可回收利用，默认 300mm |
| 已有余料 | Array | `warehouse_stock`（余料） | 仓库中已有的可用余料列表 |

### S3.3 输出结果

```json
{
  "optimizationId": "opt_001",
  "summary": {
    "totalBarsUsed": 15,
    "totalBarsLength": 90000,
    "totalCutLength": 82350,
    "totalWaste": 7650,
    "utilizationRate": 91.5,
    "reusableSurplus": [
      { "materialCode": "001", "length": 450, "count": 3 }
    ]
  },
  "cuttingPlans": [
    {
      "barIndex": 1,
      "materialCode": "001",
      "materialName": "框上横",
      "barLength": 6000,
      "isNewBar": true,
      "cuts": [
        { "length": 2390, "orderNumber": "FG2026030101", "windowCode": "C1", "quantity": 1 },
        { "length": 2390, "orderNumber": "FG2026030101", "windowCode": "C2", "quantity": 1 },
        { "length": 1190, "orderNumber": "FG2026030102", "windowCode": "C1", "quantity": 1 }
      ],
      "waste": 25,
      "surplus": 0,
      "utilizationRate": 99.6
    }
  ]

```

| 输出字段 | 说明 |
|---|---|
| `summary.totalBarsUsed` | 使用的原材料棒数 |
| `summary.utilizationRate` | 总体材料利用率（%） |
| `summary.reusableSurplus` | 可回收余料列表 |
| `cuttingPlans[]` | 每根棒料的切割方案 |
| `cuttingPlans[].cuts[]` | 该棒料上的每一刀：长度、所属订单、窗号 |
| `cuttingPlans[].waste` | 不可回收废料长度 |
| `cuttingPlans[].surplus` | 可回收余料长度 |

### S3.4 优化目标与约束

| 项目 | 定义 |
|---|---|
| **主优化目标** | 最大化材料利用率（最小化废料） |
| **次优化目标** | 最小化使用的原材料棒数 |
| **约束 1** | 每根棒料上的切割总长度 + 刀口宽度 × (切割次数-1) ≤ 棒料长度 |
| **约束 2** | 优先使用仓库余料，再使用新棒料 |
| **约束 3** | 同一订单的料尽量排在相邻位置（便于车间分拣） |
| **约束 4** | 余料长度 ≥ 余料回收阈值时，计入可回收余料 |

### S3.5 算法选型建议

| 算法 | 适用场景 | 优缺点 |
|---|---|---|
| **FFD（首次适配递减）** | 快速近似 | 速度快（O(n log n)），利用率约 85-92%，适合实时预览 |
| **BFD（最佳适配递减）** | 中等精度 | 速度较快，利用率约 88-95% |
| **列生成法（Column Generation）** | 精确最优 | 可达最优解，但计算时间较长，适合最终优化 |
| **遗传算法/模拟退火** | 大规模问题 | 近似最优，适合 1000+ 需求料的大批量场景 |

**推荐方案**：默认使用 **BFD** 算法提供快速结果，用户可选择"深度优化"模式使用**列生成法**获取最优解。

### S3.6 优化结果展示页面

URL: `/optimization/{orderId}`

**页面布局：**

| 区域 | 内容 |
|---|---|
| 顶部统计栏 | 使用棒数、总利用率、总废料长度、可回收余料数 |
| 切割图示区 | 每根棒料以水平条形图展示，不同颜色标识不同订单/窗号的切割段，灰色标识废料 |
| 切割清单表格 | 棒料序号、物料名称、棒料长度、切割明细、利用率 |
| 操作按钮 | 导出 PDF、导出 Excel、导出 CNC 指令、发送到车间 |

### S3.7 与仓库模块联动

1. 优化计算时，自动查询 `warehouse_stock` 中的余料库存。
2. 优化完成后，用户确认方案时：
   - 自动生成领料出库单（`warehouse_transaction`），扣减原材料库存。
   - 自动生成余料入库单，将可回收余料入库。
3. 余料管理页面显示所有可用余料，支持手动调整。
