## 1. 总体评估

PRD_V5_Complete.md 经过 V1→V5.2 共 6 轮迭代，已经从最初的零散文档演进为一份 3424 行的"金标准"文档。PRD_README.md 明确了单一事实来源（Single Source of Truth）原则，archive 目录归档了所有历史版本，文档治理层面做得相当规范。

但从"拿到这份文档能否直接开工"的角度审视，仍然存在若干结构性问题。以下是五个维度的综合评分：

| 维度 | 评分 | 核心判断 |
| :--- | :--- | :--- |
| **文档治理与组织** | 9.0/10 | 单一事实来源、版本历史、归档策略均到位，PRD_README 是亮点 |
| **需求完整性** | 7.5/10 | 14 个模块全覆盖，但 4 个模块（AI 炫图、阳光房 3D、AR、设备对接）仍停留在功能列表层面 |
| **技术可执行性** | 6.5/10 | 算料 DSL 有框架但缺 EBNF；画布引擎选型未锁定；下料优化仅有算法名称无伪代码 |
| **内部一致性** | 7.0/10 | V5 修复了大量 V3 的冲突，但仍有 3 处跨文档矛盾和 2 处内部矛盾 |
| **产品商业性** | 5.5/10 | 缺少竞品差异化定位、用户画像、北极星指标、MVP 边界定义 |

**一句话总结：** 文档的"骨架"已经非常完整，但"肌肉"（可执行的技术规格）和"灵魂"（产品战略）仍需补强。

## 2. 文档治理层面：做对了什么

在指出不足之前，有必要先肯定 V5 迭代中做对的几件事，因为这些是后续改进的基础：

**第一，确立了单一事实来源原则。** PRD_README.md 明确声明"开发人员请只看 `docs/PRD_V5_Complete.md` 这一个文件"，并将所有历史文档归入 `archive/`。这从根本上解决了 V3 时代"14+ 份零散文档"导致的混乱问题。在实际团队协作中，这一原则的价值怎么强调都不为过——它消除了"到底以哪份文档为准"的争论。

**第二，修订历史清晰可追溯。** 从 V1.0 到 V5.2 的每次修订都有明确的版本号、日期和变更摘要。这使得任何团队成员都能快速了解文档的演进脉络，也为后续的变更管理提供了模板。

**第三，数据模型定义的深度和广度。** 41 张数据表的完整字段定义（含类型、键、可空、默认值、描述）是整份文档中质量最高的部分。特别是 `design_data_json` 的 Schema 定义（第 4 章），从顶层结构到每个组件的字段都有 JSON 示例和取值范围说明，这对前后端开发人员来说是可以直接编码的规格。

**第四，V5.2 的画图模块微观交互定义。** 控制点交互、中梃拖拽状态机、2D→3D 映射规则、导出打印规格等内容的补充，使画图模块从"功能列表"提升到了"交互规格"的层次。

## 3. 核心短板分析（按严重程度排序）

### 3.1 短板一：产品战略层文档完全缺失（严重）

PRD_V5 直接从"产品概述"跳到了"数据模型"，中间缺少了产品文档中最关键的战略层内容。具体缺失如下：

**竞品差异化定位。** 产品定义文档.md 提到了竞品"画门窗Pro"（新格尔集团旗下），但 PRD_V5 中没有任何竞品分析。门窗设计软件市场已有多个成熟产品（画门窗Pro、门窗大师、酷家乐门窗版、FensterCAD），我们的产品与它们的差异化定位是什么？是更便宜、更易用、还是功能更全？这个问题不回答，后续所有的功能优先级排序都缺乏依据。

**目标用户画像。** PRD 2.2 节定义了三个订阅计划（试用版/门店版/工厂版），但没有回答更基本的问题：我们的核心用户是谁？是年产值 100 万以下的小门窗店？还是年产值 500 万以上的中型工厂？不同规模的企业，对功能的需求优先级完全不同——小门窗店最需要的是快速画图报价，中型工厂最需要的是生产排程和下料优化。

**北极星指标。** 文档没有定义产品成功的衡量标准。是月活跃用户数（MAU）？是付费转化率？是单用户平均收入（ARPU）？没有北极星指标，团队就无法判断哪些功能对产品成功的贡献最大。

**MVP 边界。** 14 个模块全部列入了 V1.0 的范围，但这在实际开发中是不现实的。哪些模块是 MVP 必须有的？哪些可以推迟到 V1.1 甚至 V2.0？文档没有明确回答。

> **建议：** 在 PRD_V5 的第 2 章和第 3 章之间，新增"产品战略"章节，包含竞品分析矩阵、用户画像、北极星指标、MVP 范围定义。预计工时 3-5 天。

### 3.2 短板二：算料公式引擎 DSL 缺乏可执行规格（严重）

第 8.3 节的 WindoorFormula DSL 定义是 V5.0 的重要新增内容，但从"开发人员拿到手能否直接编码"的标准来看，仍有明显差距：

**缺少正式的语法定义（EBNF/BNF）。** 当前的 DSL 规格是通过"数据类型表 + 变量表 + 运算符表 + 函数表"的方式描述的，这对理解 DSL 的能力范围是足够的，但对于实现一个解析器来说是不够的。开发人员需要知道：表达式的优先级和结合性是什么？三元运算符 `? :` 的嵌套规则是什么？`IF(cond, a, b)` 和 `cond ? a : b` 是否等价？这些问题只有通过正式的 EBNF 语法才能无歧义地回答。

**缺少错误处理规格。** 当公式中引用了不存在的变量时，引擎应该返回什么？是 0、null、还是抛出错误？当除数为 0 时如何处理？当公式语法错误时，错误信息的格式是什么？这些都是实现公式引擎时必须回答的问题。

**缺少技术选型决策。** 8.3.6 节描述了执行流程（解析→绑定→求值→校验→输出），但没有回答关键的技术选型问题：公式在前端执行还是后端执行？如果在前端执行，如何防止恶意公式（如无限循环）？用什么库来实现 AST 解析（自研 vs 使用 math.js / expr-eval 等现成库）？

**缺少性能基准。** "算料计算时间 ≤ 3 秒"是非功能性需求中的目标，但没有说明这个目标对应的场景复杂度。一个只有外框和两块玻璃的简单窗，和一个有 20 个中梃、10 个扇、格条装饰的复杂窗，算料时间差异可能是 100 倍。需要定义具体的性能测试用例。

> **建议：** 撰写独立的《算料公式引擎技术规格书》，包含 EBNF 语法定义、错误处理矩阵、技术选型 PoC 报告、性能测试用例。预计工时 5-7 天。

### 3.3 短板三：四个"空洞模块"仍未实质补齐（中等）

V3 缺失清单中的 P1-05 到 P1-08 指出了四个空洞模块：AI 炫图、阳光房 3D、AR 拍照、设备对接。V5 对这四个模块做了一定程度的补充，但从可执行性角度看，仍然不够：

| 模块 | V5 补充内容 | 仍缺失的关键内容 |
| :--- | :--- | :--- |
| **AI 炫图**（16.1 节） | 定义了输入/输出、风格选项、费用限额 | 缺少 AI 服务选型（ControlNet + SD 只是列出，未做技术可行性验证）、异步任务管理的完整设计、失败重试策略、GPU 资源规划 |
| **阳光房 3D**（16.2 节） | 定义了功能列表和基本交互 | 缺少 3D 数据模型（阳光房的 JSON Schema 是什么？与门窗 design_data_json 如何关联？）、Three.js 场景管理方案、性能优化策略 |
| **AR 预览**（16.3 节） | 定义了 5 步流程 | 缺少 AR 技术选型（WebXR vs 原生 SDK）、摄像头权限处理、透视变换算法、设备兼容性矩阵 |
| **设备对接**（16.5 节） | 定义了 CNC 对接协议和数据格式 | 缺少具体设备品牌的协议适配方案、通信中间件设计、离线缓冲策略 |

这四个模块的共同问题是：**有"做什么"的描述，但缺少"怎么做"的技术方案。** 对于 V1.0 来说，如果这四个模块不在 MVP 范围内（建议不在），那么可以暂时搁置。但如果团队计划在 V1.1 或 V2.0 中实现，就需要尽早启动技术预研。

> **建议：** 明确这四个模块的版本归属（V1.0/V1.1/V2.0），对于计划在下一个版本实现的模块，提前 2-3 个 Sprint 启动技术预研并输出 PoC 报告。

### 3.4 短板四：画布引擎技术选型未锁定（中等）

PRD 2.3 节列出了"Fabric.js / Konva.js（2D）+ Three.js（3D/阳光房）"作为画布引擎选型，但用了斜杠"/"表示"或"，说明选型尚未确定。画图模块可执行规格书中也没有给出明确的选型结论。

这是一个需要尽早锁定的决策，因为它影响：
- 前端团队的技术栈学习成本
- 画布组件的 API 设计
- 性能优化策略（Canvas 2D vs WebGL）
- 与 Three.js 3D 预览的数据共享方式

当前项目的实际实现（ARCHITECTURE.md）使用的是手写 SVG 渲染，这与 PRD 中提到的 Fabric.js/Konva.js 都不一致。这种"文档说一套、代码做一套"的情况需要尽快统一。

> **建议：** 用 1 周时间做一个画布引擎选型 PoC，对比 SVG（当前方案）、Fabric.js、Konva.js 在以下场景的表现：100+ 组件渲染性能、中梃拖拽交互流畅度、异形框（弧形/三角形）渲染能力、与 Three.js 的数据共享便利性。基于 PoC 结果锁定选型并更新 PRD。

### 3.5 短板五：跨文档矛盾仍然存在（轻微）

虽然 V5 已经大幅减少了跨文档矛盾，但仍有以下几处需要修正：

| 矛盾点 | 文档 A | 文档 B | 建议 |
| :--- | :--- | :--- | :--- |
| **订阅计划定价** | PRD 2.2 节：试用版/门店版(¥299)/工厂版(¥899)，共 3 档 | 附录 21.7 节：免费版/基础版(¥99)/专业版(¥299)/企业版(¥699)，共 4 档 | 统一为一套定价方案，删除另一套 |

## 7. 结论

PRD_V5_Complete.md 是一份质量相当高的产品需求文档。经过 6 轮迭代，它已经从最初的零散文档演进为一份 3424 行、覆盖 14 个模块、41 张数据表、23 组 API 的自包含文档。V3 缺失清单的 36 项中，97.2% 已经得到不同程度的补齐，P2 级 15 项更是 100% 完全补齐。

但从"能否直接开工"的标准来看，仍有三个关键瓶颈需要突破：

1. **产品战略层缺失** — 没有竞品分析、用户画像、北极星指标和 MVP 定义，团队缺乏优先级排序的依据
2. **算料引擎缺乏可执行规格** — DSL 有框架但无 EBNF，开发人员无法直接编码
3. **画布引擎选型未锁定** — 文档说 Fabric.js/Konva.js，代码用 SVG，需要尽快统一

建议按照第 6 节的行动计划，在 2-4 周内完成核心改进项，使文档达到"开发人员拿到手可以直接编码"的水平。
