# 《门窗之星》产品需求文档 V4.0 第二轮深度审查报告

**版本：** 1.0
**日期：** 2026年03月02日
**审查人：** Manus AI (产品经理)

---

## 1. 审查概述

本次审查的核心目标，是在 PRD V4.0 已取得显著进步的基础上，以更严苛、更贴近真实商业运营的视角，对整个文档体系进行第二轮深度考察。V4.0 成功补齐了多租户、核心算法、业务流程闭环等 36 项关键短板，使产品骨架基本成型，达到了“可开发”状态。然而，从“可开发”到“可运营”、“有竞争力”的商业化产品，仍存在若干关键差距。

本报告将 V4.0 视为一个全新的基线，重点识别那些在第一轮审查中未被充分暴露的、更深层次的结构性问题、业务逻辑盲区和商业模式缺陷。报告发现，当前文档体系虽然内容详实，但在**结构整合度、核心业务深度、企业级功能完备性**和**商业化支撑**四个维度上存在明显不足。这些不足若不加以解决，将可能导致产品在市场竞争中处于不利地位，难以满足中大型门窗企业的复杂需求。

## 2. 核心结论

PRD V4.0 已经是一个功能相对完整的门窗行业 SaaS 软件蓝图，但其本质上仍是一个**“单点工具”的集合**，而非一个真正整合的**“企业运营平台”**。产品设计的重心偏向于“画图”和“算料”这两个核心环节，而对支撑企业高效运转的**采购、生产、质量、财务、协作**等流程的理解和定义尚显单薄。

我们识别出 **23 项** 关键短板，并将其归类为五大问题域，按优先级（P0-P2）进行了划分。其中，**P0 级问题（4项）** 涉及文档结构和核心业务逻辑，不解决将导致开发混乱和产品核心功能残缺，建议立即修复。

## 3. 主要问题与改进建议

### 3.1. P0：阻塞级问题（应立即修复）

| 编号 | 问题分类 | 问题描述 | 潜在影响 | 改进建议 |
| :--- | :--- | :--- | :--- | :--- |
| P0-01 | 文档结构 | **章节编号严重冲突** | 补充章节与原文章节编号重复（18-22章），造成严重混淆，开发人员无法定位正确需求。 | **重新编排章节**：将 5 份补充文档的内容**完全整合**进 PRD 主文档，废弃 supplements 目录，重新编排所有章节号，确保唯一性和逻辑连贯性。 |
| P0-02 | 文档结构 | **补充内容未整合** | PRD V4.0 只是通过链接指向外部文件，破坏了“单一事实来源”原则，增加开发团队的理解成本和出错风险。 | **内容深度整合**：将所有补充内容按模块归属，合并到 PRD_V3.md 的相应章节中，形成一份完整、独立的 PRD V5.0。 |
| P0-03 | 业务逻辑 | **组合窗设计功能缺失** | 无法设计行业内常见的连窗、转角窗、拼樱组合窗，严重限制了产品的适用场景，无法满足大部分定制需求。 | **重构 `design_data_json`**：将其从“单窗”结构扩展为支持“画布-组合-单窗”的层级结构，并定义转角柱、拼接料等新组件及其算料规则。 |
| P0-04 | 业务逻辑 | **权限模型 (RBAC) 定义不充分** | 仅有角色名称，无完整的权限矩阵，开发无法实现精确的访问控制，导致严重安全隐患和业务流程混乱。 | **建立权限矩阵**：以表格形式，详细定义每个角色对每个模块、每个操作（增删改查、导入导出、审批）的具体权限。 |

### 3.2. P1：核心功能问题（V1.1 版本应补齐）

| 编号 | 问题分类 | 问题描述 | 潜在影响 | 改进建议 |
| :--- | :--- | :--- | :--- | :--- |
| P1-01 | 业务逻辑 | **采购与供应商管理缺失** | 无法管理供应商、下采购单、跟踪采购进度，导致产供销脱节，库存管理沦为空谈。 | 新增“采购管理”模块，包含供应商信息库、采购订单、询价/比价、采购入库等功能。 |
| P1-02 | 业务逻辑 | **BOM (物料清单) 概念缺失** | 算料结果未形成标准BOM，无法与生产、采购有效联动，数据孤岛问题严重。 | 建立标准的BOM数据模型，包含版本管理。算料引擎的输出应为结构化的BOM，并可传递给生产和采购模块。 |
| P1-03 | 业务逻辑 | **生产排程功能过于简单** | 仅有状态流转，无法满足工厂对产能、工序、交期的精细化管理需求。 | 增强“车间管理”模块，增加工序定义、产能规划、甘特图排程、交期预测等功能。 |
| P1-04 | 业务逻辑 | **质量管理 (QC) 模块缺失** | 缺乏来料检验、过程检验、成品检验流程，无法保证产品质量，缺少质量追溯能力。 | 新增“质量管理”模块，支持检验标准设定、质检任务、不合格品处理和质量追溯报表。 |
| P1-05 | 业务逻辑 | **打印/导出体系不完整** | 报价单、生产单、发货单、标签条码等关键业务单据的格式未定义，无法指导开发。 | 为每种核心单据设计详细的打印模板，明确包含的字段、布局、Logo位置等。 |
| P1-06 | 技术规格 | **`design_data_json` 版本管理缺失** | 仅提了一句“保留10个版本”，但无具体实现方案，导致版本回退、对比等功能无法开发。 | 详细设计版本存储方案（快照或Diff）、版本对比的可视化界面以及相关的API。 |
| P1-07 | 商业模式 | **订阅计划与功能权限映射缺失** | 未定义不同订阅计划（试用/门店/工厂）分别包含哪些功能，商业模式不清晰。 | 建立详细的“订阅计划-功能权限”映射表，明确各版本的功能、用量、服务差异。 |

### 3.3. P2：体验与扩展问题（V1.2 及以后版本考虑）

此部分问题虽然不影响核心流程，但对提升产品竞争力和用户体验至关重要，包括：完整的报价折扣体系、数据导入/迁移方案、独立的统计报表模块、更完善的移动端适配、面向终端客户的功能（如进度查询、在线验收）以及计费与支付集成等 **12 项**。详细列表请参见附录。

## 4. 总结与后续步骤

PRD V4.0 的文档体系已经为产品开发奠定了坚实的基础。然而，要实现从“工具”到“平台”的跨越，必须正视当前在业务深度和企业级功能上的短板。我们强烈建议，在启动大规模开发之前，产品团队应优先投入资源，**立即解决 P0 级问题**，特别是对文档结构进行重构，将所有补充内容深度整合，形成一份统一、连贯、无歧义的 PRD V5.0。在此基础上，再规划 P1 和 P2 级功能的迭代开发，稳步将《门窗之星》打造为行业领先的企业级解决方案。

### 附录：P2 级问题清单

- **报价体系**：缺少阶梯报价、批量折扣、促销活动等机制。
- **数据导入/迁移**：缺少客户、订单、型材库等历史数据的导入方案。
- **搜索功能**：缺少对搜索字段、排序规则、全文检索需求的详细定义。
- **数据报表模块**：缺少独立的报表中心，无法满足销售、生产、库存、利润等多维度分析需求。
- **文档交叉引用不一致**：补充文档之间存在定义冲突（如 `plan` 字段与用量限制）。
- **仓库与算料联动不足**：缺少采购建议、供应商管理等高级库存功能。
- **移动端适配**：除扫码外，业务员、安装师傅、管理者的移动办公需求未明确。
- **协作功能**：缺少 `@提及`、操作动态流等团队协作功能。
- **客户端功能**：缺少面向终端客户的订单进度查询、在线验收、服务评价等功能。
- **计费与支付**：缺少订阅费用的支付集成、发票管理、续费提醒等商业化闭环功能。
- **物流配送管理**：缺少发货后的物流跟踪、路线规划、签收确认等功能。
- **文档一致性**：附录中的状态机图未根据补充内容同步更新。
