## 3. 结构性问题（3 类）

### 3.1 版本标注三方矛盾

这是当前文档体系中最严重的结构性问题。同一份文档在三个位置标注了三个不同的版本号：

| 位置 | 标注内容 | 实际应为 |
| :--- | :--- | :--- |
| `PRD_README.md` 第 11 行 | "V5.3，3500+ 行" | 正确 |
| `docs/PRD_V5_Complete.md` 第 3 行（文件头部） | "版本：V5.3 Complete" | 正确 |
| `docs/PRD_V5_Complete.md` 第 3592 行（文件尾部） | "文档版本：**V5.2** Complete" | **错误**，应为 V5.3 |
| `docs/README.md` 第 1 行 | "文档版本：**V5.0** Complete" | **错误**，应为 V5.3 |
| `docs/README.md` 第 9 行 | "3270 行" | **错误**，实际 3599 行 |

**影响：** 开发人员如果从 `docs/README.md` 进入（而非从 `PRD_README.md`），会误以为文档仍是 V5.0 版本，缺少 V5.1-V5.3 的全部增强内容。这直接违背了"单一事实来源"原则。

**修复建议：** 统一所有版本标注为 V5.3，更新 `docs/README.md` 的版本号和行数，修复 `PRD_V5_Complete.md` 尾部的版本标注。工时估算：0.5 小时。

## 4. 内容短板（9 项）

### 4.1 技术栈三方冲突（严重）

这是影响开发启动的最关键问题。三份文档对技术栈的描述互相矛盾：

| 技术点 | PRD_V5_Complete.md（V5.3） | ARCHITECTURE.md | 实际代码 |
| :--- | :--- | :--- | :--- |
| React 版本 | React 18 | React 19 | React 19 |
| 状态管理 | Zustand | useReducer（DesignerContext） | useReducer |
| 画布引擎 | Konva.js（2D）+ Three.js（3D） | 手写 SVG + Three.js | 手写 SVG + Three.js |
| CSS 框架 | TailwindCSS（无版本号） | Tailwind 4 | Tailwind 4 |
| 后端框架 | NestJS + Prisma | 未定义（纯前端项目） | 无后端 |
| 数据库 | MySQL 8.0 + Redis | 未定义 | 无数据库 |

**核心矛盾：** PRD 明确写了"Konva.js"作为 2D 画布引擎，但实际代码使用手写 SVG。这不是一个可以忽略的差异——Konva.js 是一个基于 Canvas 2D 的库，而 SVG 是完全不同的渲染范式，两者的性能特征、事件处理模型和组件架构都截然不同。如果团队按 PRD 的 Konva.js 方案开发，现有的 SVG 代码需要全部重写。

**修复建议：** 必须在开发启动前做出明确决策并统一所有文档。建议做一个为期 3 天的 PoC（概念验证），对比 SVG、Konva.js 和 Fabric.js 在以下场景的表现：100+ 组件的渲染帧率、中梃拖拽的交互流畅度、异形窗（弧形）的渲染能力。PoC 结论写入 `ARCHITECTURE.md` 并同步更新 PRD。工时估算：3 天。

### 4.2 画图模块可执行规格书与主文档的数据模型冲突（严重）

`docs/画图模块_可执行规格书.md`（429 行）被 `docs/README.md` 列为"参考文档，面向前端开发"。但该文档定义的数据模型与 `PRD_V5_Complete.md` 第 4 章（画布数据 Schema）存在根本性差异：

| 概念 | 可执行规格书 | PRD_V5 第 4 章 |
| :--- | :--- | :--- |
| 顶层对象 | `WindowUnit`（含 `Frame` → `Opening` 递归树） | `design_data_json`（含 `canvas`、`frame`、`mullions`、`sashes`、`glasses` 平铺） |
| 数据结构 | **递归树形**：`Opening.childOpenings[]` | **扁平数组**：`mullions[]`、`sashes[]`、`glasses[]` 各自独立 |
| 扇类型枚举 | `SashOpeningType`（14 种，含 `tilt_and_turn_inward_left`） | `sash.openDirection`（10 种，含 `左内开`） |
| 五金件 | `Hardware` 接口（type + model + position） | 未在 Schema 中定义 |
| 型材系列 | `ProfileSeries` 接口（含 formulas 对象） | 独立的 `profile_series` 数据库表 |

**影响：** 前端开发人员如果参考可执行规格书的 TypeScript 接口来编码，写出的代码将无法与 PRD 定义的后端 API 对接。这两份文档必须统一。

**修复建议：** 以 PRD_V5 第 4 章的 `design_data_json` 为准（因为它与后端存储和 API 对齐），更新可执行规格书的数据模型部分使其一致。如果可执行规格书的递归树形结构确实更优，则应反向更新 PRD。无论哪个方向，两份文档的数据模型必须完全一致。工时估算：1 天。

| API 组 | 路径数 | 有请求/响应示例 | 缺失内容 |

## 5. V5.3 新增内容质量评估

V5.3 版本新增了 4 个方面的内容。以下是对每个新增内容的质量评估：

### 5.1 产品战略（评分：8/10）

竞品分析覆盖了 4 家主要竞品，SWOT 分析到位，差异化定位清晰（"高性价比的一体化"）。用户画像定义了两个典型角色，痛点和期望描述具体。北极星指标选择合理（有效订单创建数），直接关联核心价值。MVP 边界定义清晰，In-Scope 和 Out-of-Scope 的划分合理。

**不足之处：** 竞品分析缺少定量数据（如各竞品的用户量、市场份额、定价细节）。用户画像缺少用户旅程图（User Journey Map）。北极星指标缺少分解指标（Leading Indicators），如"周活跃设计数"、"算料完成率"等。

### 5.2 算料引擎增强（评分：8.5/10）

EBNF 语法定义清晰，覆盖了表达式、函数调用、变量引用等核心语法。7 种错误码的分类合理（语法错误、引用错误、求值错误），前端提示示例实用。安全沙箱规格明确（Web Worker + 白名单 + 超时限制）。执行流程的 5 个阶段（解析→绑定→求值→校验→输出）定义清晰。

**不足之处：** EBNF 缺少对三元条件表达式（`? :`）、比较运算符（`>`、`<`）和逻辑运算符（`&&`、`||`）的语法定义，但这些运算符在第 8.3.3 节被列为支持的运算符。这意味着 EBNF 与运算符表不一致。此外，缺少公式编辑器的前端交互规格（语法高亮、自动补全、实时校验的具体方案）。

### 5.3 一致性修复（评分：7/10）

订阅定价统一为三档，消除了之前的矛盾。数据模型统一规范（主键、外键、状态字段、布尔值）为后端开发提供了明确的编码标准。

**不足之处：** 技术栈统一写了"Konva.js + NestJS + Prisma"，但实际代码使用 SVG + React useReducer，引入了新的矛盾（见 4.1 节）。此外，V5.3 一致性修复文档（`V5_3_Consistency_Fix.md`）中的主键规范写"统一为 `id`，类型为 `VARCHAR(36)`"，但主文档第 3.43 节写"统一为 `xxx_id`"——两者不一致。

### 5.4 工程化补充（评分：8/10）

数据库索引设计覆盖了 11 张核心表，复合索引的设计考虑了实际查询场景。API 限流策略按订阅计划分级，特殊接口（登录、AI 渲染、文件上传）有独立限流规则。API 版本控制策略（URL 路径 + 6 个月废弃期）符合行业最佳实践。压测方案定义了 5 种场景和明确的通过标准。

**不足之处：** 索引设计只覆盖了 41 张表中的 11 张。压测方案中的并发数（50/200 用户）与非功能性需求中的目标（500 同时在线）存在差距。

## 7. 总结

V5.3 版本的 PRD 在内容深度和广度上已经达到了一个相当高的水平——3599 行、21 章、14 个功能模块、41 张数据表、23 组 API，覆盖了从设计到生产的全链路。产品战略层的补强（竞品分析、用户画像、北极星指标、MVP 边界）使文档从"技术规格书"升级为"产品规格书"。算料引擎的 EBNF 定义和安全沙箱规格为核心技术模块提供了可执行的实现指南。

但文档体系的"最后一公里"问题仍然突出：版本标注的三方矛盾、技术栈的三方冲突、画图模块两份文档的数据模型不一致——这三个问题如果不在开发启动前解决，将直接导致团队在技术选型上摇摆不定，前后端在数据结构上各说各话。

**建议的优先级排序：** 先花 1 天做 A1-A5（文档清理），再花 3 天做 B1（画布引擎 PoC），然后 1 天做 B2（数据模型统一）。这 5 天的投入可以消除 80% 的结构性风险，让团队在第 2 周就能全速进入编码阶段。
