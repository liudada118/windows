## 八、关于 PRD 拆分的思考

当前 PRD_V5_Complete.md 采用"单一事实来源"策略，将所有内容集中在一个 4613 行的文件中。这个策略在 V5.0 时期（3270 行）是合理的，但经过 V5.1-V5.4 四轮追加式编辑后，已经出现了结构腐化的迹象。

**单文件策略的优势：**
- 搜索方便（Ctrl+F 即可全文检索）
- 不存在跨文件引用断裂的风险
- 版本管理简单（一个文件一个版本号）

**单文件策略的劣势（已显现）：**
- 追加式编辑容易导致章节错位和重复
- 4600+ 行的文件在 GitHub 上渲染缓慢
- 多人同时编辑同一文件容易产生合并冲突
- 不同角色（前端/后端/产品）需要关注的章节差异很大

**建议的折中方案：**

保持 PRD_V5_Complete.md 作为"编译产物"（类似代码的 build 产物），但日常维护改为按模块拆分的源文件：

```
docs/
  prd/
    00-overview.md          # 第 1-2 章：修订历史、产品概述
    01-data-model.md        # 第 3 章：数据模型
    02-canvas-schema.md     # 第 4 章：画布数据 Schema
    03-auth.md              # 第 5 章：认证与权限
    04-order.md             # 第 6 章：订单管理
    05-drawing.md           # 第 7 章：画图核心
    06-formula.md           # 第 8 章：算料引擎
    07-production.md        # 第 9 章：生产管理
    ...
    14-api.md               # 第 19 章：API 定义
    15-nfr.md               # 第 20 章：非功能性需求
    16-appendix.md          # 第 21 章：附录
  PRD_V5_Complete.md        # 自动拼接生成的完整版（只读）
  build-prd.sh              # 拼接脚本
```

这样每个模块的负责人只需编辑自己的文件，通过脚本自动拼接为完整版，避免追加式编辑导致的结构腐化。
