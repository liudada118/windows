# 其他缺失项补充

## 3.3. 数据库索引设计

**原则:** 为所有外键、高频查询条件的字段、以及用于排序的字段建立索引。

| 表名 | 建议索引 | 类型 |
| :--- | :--- | :--- |
| `orders` | `tenant_id`, `user_id`, `status`, `created_at` | 普通索引 |
| `order_items` | `order_id`, `product_id` | 普通索引 |
| `users` | `tenant_id`, `phone` | 唯一索引 |
| `customers` | `tenant_id`, `phone` | 唯一索引 |

## 19.1.3. API 限流策略

采用基于令牌桶算法的速率限制。

| 用户类型 | 限制范围 | 速率 |
| :--- | :--- | :--- |
| 未登录用户 | 按 IP 地址 | 100 请求 / 分钟 |
| 普通用户 | 按用户 ID | 500 请求 / 分钟 |
| 企业版用户 | 按租户 ID | 2000 请求 / 分钟 |

- **超出处理:** 返回 `429 Too Many Requests` HTTP 状态码，并在响应头中包含 `Retry-After` 字段，告知客户端何时可以重试。

## 20.10. 压力测试方案

- **目标:** 模拟 1000 用户同时在线，其中 20% 的用户（200人）在进行高频的画图和算料操作。
- **工具:** k6 (Grafana k6)
- **场景:** 
  1. **登录与验证:** 1000 用户在 1 分钟内完成登录。
  2. **常规操作:** 800 用户以 10秒/次的频率随机浏览订单、客户列表。
  3. **核心操作:** 200 用户以 5秒/次的频率执行“创建窗户 → 修改尺寸 → 点击算料”的循环操作。
- **验收标准:**
  - **P95 响应时间:** 所有 API < 500ms。
  - **错误率:** < 0.1%。
  - **CPU/内存使用率:** 压测期间，服务器 < 80%。
