## 1. 项目概述

### 1.1. 项目背景

当前中小型门窗企业普遍面临设计效率低、报价算料易出错、生产管理混乱、前后端信息脱节等痛点。传统依赖人工CAD绘图和Excel表格管理的模式，已无法满足市场对快速响应和精细化生产的要求。本项目旨在复刻并优化“画门窗”(WindoorCraft)软件的核心功能，打造一款集设计、报价、拆单、生产管理于一体的SaaS解决方案，帮助门窗企业实现全业务流程的数字化转型。

### 1.2. 产品愿景

我们的愿景是“让天下没有难做的门窗生意”。通过提供一款极简易用、功能强大且价格亲民的专业软件，赋能全球中小型门窗门店和工厂，使其能够轻松应对复杂的业务挑战，提升核心竞争力。

### 1.3. 目标用户

本产品主要服务于全球范围内的门窗行业从业者，核心用户画像如下：

| 用户角色 | 核心诉求 |
| :--- | :--- |
| **门窗门店销售/设计师** | 快速为客户设计方案、提供逼真效果图、精准报价、简化下单流程。 |
| **门窗工厂管理员/拆单员** | 准确无误地接收门店订单、快速完成拆单算料、优化型材使用、安排生产计划。 |
| **企业主/管理者** | 实时掌握订单状态、监控生产进度、管理客户与财务、降低运营成本。 |

## 2. 业务流程

软件需支撑从门店接单到工厂交付的完整业务闭环。核心流程如下：

**门店端 (销售与设计)**
1.  **客户沟通**: 销售人员与客户沟通需求。
2.  **方案设计**: 使用软件的2D/3D编辑器快速绘制窗型图。
3.  **效果展示**: 生成AI效果图或3D动画，向客户展示安装效果。
4.  **报价与修改**: 根据设计方案自动生成报价单，并根据客户反馈进行修改。
5.  **签约下单**: 客户确认方案后，通过电子签名完成签约，订单一键提交至工厂端。

**工厂端 (生产与管理)**
1.  **订单接收**: 系统自动接收门店提交的订单，进入待处理列表。
2.  **拆单算料**: 拆单员对订单进行审核，系统自动完成拆单，生成型材、玻璃、五金等物料清单。
3.  **型材优化**: 系统自动进行型材优化排版，生成优化切割方案和领料单。
4.  **生产排产**: 管理员根据订单优先级和产能情况，下达生产任务单至车间。
5.  **生产执行与追溯**: 工人通过扫描任务单上的二维码进行生产，并通过手持设备报工，管理者可实时追踪生产进度。
6.  **设备对接**: 生产数据可直接下发至双头锯、加工中心等数控设备。
7.  **成品入库与发货**: 生产完成的成品扫码入库，发货时通过盘点机防止错发漏发。

## 3. 产品功能规划

### 3.1. 产品架构 (功能模块)

产品将采用模块化设计，主要分为三大核心层：**基础数据层**、**核心业务层** 和 **终端应用层**。

- **基础数据层**: 作为系统的基石，管理所有业务所需的基础信息，包括产品、物料、客户、员工等。
- **核心业务层**: 包含软件的核心功能模块，如设计绘图、订单管理、生产管理等。
- **终端应用层**: 面向不同用户角色的客户端，提供跨平台（Web/PC/Mobile）的一致体验。

## 4. 非功能性需求

- **性能**: 2D编辑器操作流畅无卡顿；3D模型秒级生成；AI渲染时间控制在30秒以内。
- **可用性**: 核心绘图操作无需培训即可上手，符合“傻瓜式”操作的定位。
- **兼容性**: 客户端支持主流浏览器（Chrome, Firefox, Safari）、Windows 7/10/11、macOS、iOS和Android。
- **数据安全**: 用户数据物理隔离，数据库加密存储，网络传输全程SSL加密。
- **国际化**: 系统需支持多语言，初期支持简体中文和英语，后续可扩展俄语、越南语等。

## 5. 待定与风险

- **核心算法风险**: 型材优化算法和异形窗的结构化计算是本项目的技术难点，需要投入核心研发力量攻关。
- **数据标准**: 门窗行业型材、五金种类繁多，没有统一标准。初期需要投入人力帮助种子客户完成产品系列的参数化建库工作。
- **市场竞争**: 市场上已存在成熟竞品，产品需要有明确的差异化优势（如更优的性价比、更极致的易用性、更开放的生态）。

## 6. 附录：2D绘图编辑器 - 业务逻辑与数据模型详述

本文档旨在对产品功能需求文档（PRD）中的“设计与报价”模块进行补充，详细阐述2D绘图编辑器的核心业务逻辑、数据模型和交互流程，以确保开发团队能够清晰、准确地理解和实现相关功能。

---

### **6.1. 核心数据模型**

编辑器的所有操作都围绕一个结构化的JSON数据对象展开。这个对象描述了门窗的每一个组成部分及其相互关系。一个简化的数据模型示例如下：

```json
{
  "id": "window_01",
  "width": 2400,
  "height": 1800,
  "profile_series_id": "series_70",
  "frame": {
    "id": "frame_01",
    "shape": "rectangle", // 'rectangle', 'arc_top', 'custom_polygon'
    "points": [{"x":0, "y":0}, {"x":2400, "y":0}, {"x":2400, "y":1800}, {"x":0, "y":1800}],
    "openings": [
      {
        "id": "opening_01",
        "rect": {"x": 50, "y": 50, "width": 2300, "height": 1700},
        "mullions": [
          {
            "id": "mullion_01",
            "type": "vertical",
            "position": 1150, // 相对于opening的x坐标
            "is_arc": false
          }
        ],
        "sashes": [], // 如果分格后没有继续嵌套，则直接填充扇
        "child_openings": [
          {
            "id": "opening_01_1",
            "rect": {"x": 50, "y": 50, "width": 1100, "height": 1700},
            "sashes": [/* 扇对象 */]
          },
          {
            "id": "opening_01_2",
            "rect": {"x": 1200, "y": 50, "width": 1100, "height": 1700},
            "sashes": [/* 扇对象 */]
          }
        ]
      }
    ]
  }
}
```

**核心对象定义:**

| 对象 | 描述 |
| :--- | :--- |
| **Window** | 整个门窗单元的顶层对象，包含尺寸、型材系列等全局信息。 |
| **Frame** | 门窗的最外层框架，由一个或多个型材段构成，定义了门窗的整体轮廓。 |
| **Opening** | “开口”或“分格”，是框架内可以被填充的区域。它可以被中梃/横档进一步分割，或者直接填充扇。 |
| **Mullion/Transom** | 中梃（垂直）和横档（水平），用于分割`Opening`的型材杆件。 |
| **Sash** | “扇”，填充在`Opening`内的具体单元，可以是固定的（Fixed）或可开启的（Operable）。扇本身也由一个小的框架和玻璃组成。 |
| **Glass** | 玻璃单元，具有厚度、类型（中空、夹胶）等属性。 |
| **Hardware** | 五金件，如执手、铰链等，与`Sash`关联。 |

---

### **6.2. 核心绘图业务逻辑**

#### **6.2.1. 创建新窗（From Scratch）**

1.  **用户操作**: 用户在画布上按下鼠标左键并拖拽，绘制一个矩形。
2.  **系统逻辑**:
    *   实时显示矩形尺寸。
    *   鼠标释放时，系统创建一个`Window`对象，其`width`和`height`由用户拖拽的矩形决定。
    *   同时创建一个与`Window`等大的`Frame`对象和一个填充整个`Frame`的顶级`Opening`对象。
    *   在画布上渲染出这个基础外框。

#### **6.2.2. 分割画布（添加中梃/横档）**

1.  **用户操作**: 鼠标悬停在某个`Opening`区域内，光标变为分割预览状态。从`Opening`的一条边拖拽到另一条相对的边。
2.  **系统逻辑**:
    *   **判断方向**: 根据拖拽的主要位移是水平还是垂直，判断用户意图是添加横档（Transom）还是中梃（Mullion）。
    *   **创建对象**: 在目标`Opening`的`mullions`或`transoms`数组中添加一个新对象，记录其位置（相对于`Opening`的坐标）。
    *   **更新分格**: 将原`Opening`标记为已分割，并根据`Mullion/Transom`的位置，创建两个新的`child_openings`子对象，重新计算它们的`rect`（位置和尺寸）。
    *   **重新渲染**: 清空并重绘该区域，画出新的中梃/横档和两个新的子分格。

#### **6.2.3. 添加扇（Sash）**

1.  **用户操作**: 从左侧组件面板拖拽一个“扇”组件（如平开扇、推拉扇）到画布上一个未被分割的`Opening`区域内。
2.  **系统逻辑**:
    *   **目标检测**: 判断鼠标释放时所在的`Opening`对象。
    *   **创建对象**: 在该`Opening`的`sashes`数组中添加一个`Sash`对象。
    *   **属性继承**: `Sash`的尺寸默认继承自其所在的`Opening`。
    *   **渲染扇**: 在画布上根据`Sash`的类型（平开、推拉等）绘制其图形表示，例如平开扇的开启方向线。

#### **6.2.4. 拖拽变形（Drag & Drop）**

这是编辑器的核心交互，其逻辑最为复杂。

1.  **拖拽框架 (Drag Frame)**:
    *   **用户操作**: 拖拽`Frame`的边或角点。
    *   **系统逻辑**: 更新`Frame`的`points`数组。然后，**递归地**、按比例地更新所有内部`Opening`、`Mullion`、`Sash`的位置和尺寸。这是一个“牵一发而动全身”的过程，所有内部元素的定位都应是相对于其父容器的。

2.  **拖拽中梃/横档 (Drag Mullion/Transom)**:
    *   **用户操作**: 拖拽画布上已存在的中梃/横档。
    *   **系统逻辑**: 更新该`Mullion/Transom`对象在其父`Opening`中的`position`属性。然后，重新计算受其影响的两个相邻`child_openings`的`rect`。再递归更新这两个`child_openings`内部所有元素的尺寸和位置。

3.  **拖拽成弧形 (Arc Mullion)**:
    *   **用户操作**: 按住特定键（如Alt）拖拽一个直线中梃的中心点。
    *   **系统逻辑**: 将该`Mullion`对象的`is_arc`属性设为`true`，并增加`arc_height`属性来记录弧高。渲染时，将直线渲染为曲线（如贝塞尔曲线）。这会使其相邻的`Opening`变为异形，需要用多边形或路径来描述其边界。

### **6.3. 交互流程细化**

| 交互动作 | 系统反馈与处理逻辑 |
| :--- | :--- |
| **选择组件** | 点击画布上的任一元素（外框、中梃、扇），该元素高亮显示，并在右侧属性面板显示其可编辑参数（如尺寸、型材、开启方式等）。 |
| **修改属性** | 在右侧属性面板修改参数后，系统实时更新对应的数据模型对象，并重绘画布以反映变化。 |
| **尺寸标注** | 尺寸标注是动态生成的，不存储在核心数据模型中。每次重绘时，系统会遍历所有主要元素（Frame, Opening, Sash），根据其`rect`或`points`属性自动生成并绘制尺寸线和数值。 |
| **撤销/重做** | 每次成功的用户操作后，将当前完整的`Window`数据模型对象（JSON字符串）压入一个“历史记录”栈。执行“撤销”时，从栈中弹出上一个状态并重新渲染；“重做”则从一个“未来记录”栈中恢复。 |

通过以上对数据模型、核心业务逻辑和交互流程的详细定义，开发团队可以构建一个健壮、可扩展且符合用户直觉的2D门窗绘图编辑器。
