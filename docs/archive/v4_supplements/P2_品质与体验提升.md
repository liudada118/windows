## S17. API 设计补全与规范

### S17.1 API 版本控制策略

- **策略**：采用 **URL 路径版本控制**。所有 API 必须以 `/api/v1/` 作为前缀。
- **示例**：`GET /api/v1/orders`
- **目的**：当未来发生不兼容的 API 变更时，可以推出 `/api/v2/`，实现平滑过渡，而不会破坏现有客户端的集成。

### S17.2 统一文件上传 API

- **接口**：`POST /api/v1/upload`
- **认证**：需要用户登录凭证 (JWT)。
- **请求**：使用 `multipart/form-data` 格式。
    - `file`: 文件内容。
    - `category`: 文件用途分类，`ENUM('avatar', 'scene', 'product_image', 'proof')`。用于决定存储路径和后续处理。
- **处理流程**：
    1. 后端接收文件。
    2. 根据 `category` 和 `tenant_id` 生成唯一的存储路径，如 `s3://bucket/tenants/{tenant_id}/avatars/{uuid}.jpg`。
    3. 将文件上传至对象存储服务（OSS/S3）。
    4. 返回文件的公开访问 URL。
- **响应**：
```json
{
  "url": "https://cdn.windoorcraft.com/tenants/.../....jpg"
}
```

### S17.3 批量操作 API 示例

为提升操作效率，针对列表页中的常见批量操作，需提供专用 API。

#### POST `/api/v1/orders/batch-confirm`

- **用途**：批量将意向单转为合同单。
- **请求体**：
```json
{
  "orderIds": ["uuid1", "uuid2", "uuid3"]
}
```
- **响应**：
```json
{
  "successCount": 3,
  "failedCount": 0,
  "errors": []
}
```

#### POST `/api/v1/materials/batch-delete`

- **用途**：批量软删除物料。
- **请求体**：
```json
{
  "materialIds": ["uuid1", "uuid2", "uuid3"]
}
```

## S18. 用户体验 (UX) 设计规格

### S18.1 异常状态处理 (Unhappy Path)

系统必须优雅地处理各种异常情况，提供明确的用户反馈。

| 异常场景 | 处理方案 | UI 反馈 |
| :--- | :--- | :--- |
| **网络断开/请求超时** | - 前端应有全局的 API 请求拦截器。<br>- 自动重试 2 次，若仍失败则提示用户。 | - 页面顶部出现提示条：“网络连接已断开，请检查网络后重试”。<br>- 正在操作的按钮显示 loading 状态，然后恢复。 |
| **保存失败 (服务端错误)** | - 后端返回 5xx 错误码。<br>- 前端捕获错误，提示用户。 | - 弹出 Modal 对话框：“操作失败，服务器发生未知错误，请稍后重试或联系客服。” |
| **并发编辑冲突** | - 采用乐观锁机制，实体表增加 `version` 字段。<br>- 用户提交更新时，检查 `version` 是否匹配。 | - 弹出 Modal：“内容已被他人修改，请刷新页面查看最新版本后再编辑。” |
| **权限不足** | - 后端返回 403 Forbidden 错误码。 | - 隐藏或禁用无权限的操作按钮。<br>- 若用户通过 URL 强制访问，显示一个标准的 403 页面：“抱歉，您没有权限访问此页面。” |

### S18.2 空状态设计 (Empty State)

所有列表、表格、看板等容器在没有数据时，不能只显示空白，必须提供引导性的空状态。

- **通用结构**：一个图标 + 一句标题 + 一段描述 + 一个主操作按钮。
- **示例 (订单列表为空)**：
    - **图标**：一个放大的订单图标。
    - **标题**：`还没有任何订单`
    - **描述**：`开始创建您的第一张门窗订单吧！`
    - **按钮**：`[+ 新建订单]` (跳转到画图页面)

### S18.3 加载状态设计 (Loading State)

- **页面级加载**：首次进入一个复杂页面（如画图页、报表页）时，显示页面骨架屏 (Skeleton Screen)，模拟页面内容的轮廓，提升感知性能。
- **组件级加载**：在页面内进行局部数据请求时（如切换 Tab、点击加载更多），在相应组件区域显示加载指示器（如 Spinner）。
- **操作加载**：点击“保存”、“生成”等按钮时，按钮应变为 loading 状态（如显示菊花图并禁用点击），防止重复提交。

### S18.4 画布业务约束规则

为防止用户设计出不符合生产实际的门窗，画图模块必须增加实时业务约束校验。

| 约束类型 | 规则示例 | 校验时机 | 反馈方式 |
| :--- | :--- | :--- | :--- |
| **尺寸约束** | - 单扇平开扇宽度不得超过 800mm。<br>- 玻璃单块面积不得超过 2.5㎡。 | 拖拽调整尺寸时 | - 超过阈值时，停止放大/缩小。<br>- 在尺寸标注旁边显示红色警告图标和提示文字。 |
| **组件关系约束** | - 中梃之间的最小间距为 100mm。<br>- 推拉扇必须成对出现。 | 添加/移动组件时 | - 无法将组件放置在无效位置。<br>- 尝试放置时，显示“间距过小”等提示。 |
| **开启约束** | - 内开扇不能与内开纱窗同时存在于同一区域。 | 添加扇或纱窗时 | - 在组件选择器中直接禁用无效的选项。<br>- 尝试添加时，提示“开启方式冲突”。 |

### S18.5 新手引导 (Onboarding)

- **首次登录引导**：用户首次登录系统时，弹出一个欢迎对话框，并提供一个分步引导 (Step-by-Step Tour) 来介绍核心功能区（如画图、订单、型材）。
- **功能引导**：对于画图等复杂模块，首次使用时，通过高亮和浮动提示框，引导用户完成“画一个窗 -> 添加一个扇 -> 计算报价”的最小闭环。
- **帮助文档入口**：在页面右上角提供一个常驻的“帮助中心”入口，链接到详细的图文/视频教程。

### S18.6 离线/弱网处理

- **画布数据本地缓存**：在画图时，所有操作应实时保存到浏览器的 `localStorage` 或 `IndexedDB` 中。
- **自动保存**：系统每隔 30 秒自动将本地缓存同步到服务器。页面标题栏显示“已保存”或“正在保存...”状态。
- **断网恢复**：如果网络断开，允许用户继续在离线状态下画图。当网络恢复时，系统自动同步本地未保存的更改。如果发生冲突，提示用户选择保留服务器版本还是本地版本。

## S19. 非功能性需求补充

### S19.1 系统可用性 (SLA)

- **目标**：核心服务（画图、订单、算料）的可用性目标为 **99.9%**。
- **计算公式**：(总时间 - 停机时间) / 总时间 × 100%。
- **停机时间定义**：因系统自身问题导致用户无法访问核心功能的分钟数，不包括计划内维护。
- **降级策略**：在极端故障情况下（如数据库主库宕机），系统应能自动降级为只读模式，保证用户仍可查看设计和订单，但不能修改。

### S19.2 监控与告警

| 监控类别 | 监控项 | 工具/方案 |
| :--- | :--- | :--- |
| **应用性能监控 (APM)** | - API 平均响应时间、P95/P99 响应时间<br>- API 错误率 (4xx, 5xx)<br>- 数据库查询性能 | - Prometheus + Grafana<br>- Sentry / DataDog |
| **业务指标监控** | - 每分钟订单创建数<br>- 每小时算料请求数<br>- 在线用户数 | - Prometheus + Grafana |
| **日志监控** | - 关键错误日志 (Error, Fatal) | - ELK Stack (Elasticsearch, Logstash, Kibana) |
| **告警** | - API 错误率 > 1% 持续 5 分钟<br>- 数据库 CPU 使用率 > 80%<br>- 关键服务进程崩溃 | - Alertmanager + PagerDuty / 钉钉/飞书机器人 |

### S19.3 国际化 (i18n) 预留

虽然 V1 版本仅支持中文，但架构上必须为未来支持多语言做好预留。

- **前端**：所有 UI 文本（按钮、标签、提示）都不能硬编码，必须使用 i18n 库（如 `react-i18next`）通过 key 来引用。例如 `t('buttons.save')`。
- **后端**：
    - API 错误信息应返回错误码和英文错误信息，由前端根据错误码和当前语言显示对应的翻译文本。
    - 数据模型中，需要支持多语言的字段（如物料名称 `material.name`），应设计为 JSON 类型，如 `name_i18n: {"en_US": "Profile", "zh_CN": "型材"}`。
- **API 协议**：API 应支持 `Accept-Language` 请求头，以便返回对应语言的数据。

### S19.4 并发指标依据

原 PRD 中提出的“支持 500 并发”指标缺乏推导依据，现补充如下：

- **用户画像**：假设目标客户为中小型门窗厂，平均员工数为 20 人，其中 5 人为活跃软件用户（设计师、业务员）。
- **市场目标**：第一年发展 100 家付费企业客户。
- **总用户数**：100 家 × 5 人/家 = 500 活跃用户。
- **并发率**：根据 SaaS 行业经验，日活用户的并发率通常在 10% - 20% 之间。
- **并发用户数 (Concurrent Users)**：500 活跃用户 × 20% = 100 并发用户。
- **结论**：系统初期应至少能支持 **100 个并发用户**的平稳访问。原定的 500 并发可作为远期扩展目标。
- **压测方案**：使用 `k6` 或 `JMeter` 等工具，模拟 100 个用户同时进行画图、算料、下单等混合场景操作，测试系统在目标负载下的响应时间和错误率。
