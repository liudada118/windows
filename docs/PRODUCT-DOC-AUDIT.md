## 一、总体评价

### 1.1 文档体系成熟度评分

| 维度 | 评分（10分制） | 说明 |
| :--- | :--- | :--- |
| **需求完整性** | 7.5 / 10 | V5 覆盖了 14 个功能模块、41 张数据表、23 组 API，覆盖面广。但多个模块停留在"功能列表"层面，缺乏交互细节 |
| **技术可执行性** | 6.0 / 10 | 数据模型和 API 定义较完整，但核心算料引擎的公式 DSL 仍是"框架"而非"规格"，开发拿到手无法直接编码 |

**综合评价：** 文档体系的**广度**已经足够，14 个模块基本覆盖了门窗厂全流程。但**深度**参差不齐——订单管理和画图模块写得较细，而算料引擎、下料优化、AI 渲染等核心差异化模块仍停留在概念层面。如果现在直接交给团队开发，**P0 级阻塞问题已在 V5 中部分补齐**（如多租户隔离、订单双状态、异常处理等），但仍有多处关键缺口需要填补。

## 二、P0 级问题详细分析（阻塞开发的硬伤）

### 2.1 算料公式引擎：有框架无规格

**现状：** V5 第 7.8 节新增了"算料引擎 DSL"章节，定义了变量体系（`W`/`H`/`FW` 等 30+ 个变量）、运算符、内置函数和条件表达式语法。这比 V3 有了质的飞跃。

**仍然缺失的关键内容：**

| 缺失项 | 影响 | 建议 |
| :--- | :--- | :--- |
| **公式语法的形式化定义（BNF/EBNF）** | 开发者无法判断 `IF(W > 1500, 2, 1)` 和 `W > 1500 ? 2 : 1` 哪种是合法语法 | 撰写严格的 EBNF 语法文档，明确每种表达式的优先级和结合性 |
| **公式执行引擎的技术选型** | V5 只说"前端执行"，但未明确用什么解析器（math.js？PEG.js？手写递归下降？） | 做 PoC 对比 3 种方案，给出推荐 |
| **公式沙箱安全策略** | 用户自定义公式可能包含恶意代码（`while(true){}`） | 定义执行超时、禁用函数白名单、内存限制 |
| **公式版本管理** | 型材系列的公式修改后，已保存的算料结果是否需要重算？ | 定义公式版本号机制和"重算"触发策略 |
| **公式错误的用户体验** | V5 只说"红色高亮错误行"，但未定义错误分类（语法错误 vs 运行时错误 vs 约束违反） | 定义 3 级错误分类和对应的 UI 反馈 |

**风险评估：** 算料引擎是整个产品的核心竞争力。如果公式规格不够严谨，开发过程中会反复返工。**建议在编码前投入 3-5 天做完整的规格书 + PoC 验证。**

### 2.2 下料优化算法：仍然完全缺失

**现状：** V5 全文搜索"下料优化"，仅在第 17.1 节报表中心提到"下料利用率报表"，但**没有任何章节定义下料优化算法本身**。

这是一个严重遗漏。下料优化（一维切割问题）是门窗行业的核心痛点——一根 6 米铝材如何切割出多根不同长度的型材，使废料最少。这个问题直接影响门窗厂的材料成本（通常占总成本 40-60%）。

**需要补充的内容：**

| 内容 | 说明 |
| :--- | :--- |
| 优化目标定义 | 最高材料利用率（目标 ≥ 92%）vs 最少切割次数 vs 最少原材料使用量 |
| 约束条件 | 原材料标准长度（6000mm）、锯口宽度（3-5mm）、余料回收阈值（≥ 300mm 可回收） |
| 算法选型 | FFD（首次适应递减）保底 + 分支定界精确求解 + 可选遗传算法 |
| 性能要求 | 1000 根型材的优化方案 < 2 秒 |
| 结果展示 | 切割方案可视化（每根原材料的切割图示）+ 利用率统计 |
| 导出格式 | PDF 切割图 + Excel 明细 + 可选 CNC G-Code |

### 2.3 数据隔离方案：已补齐但有漏洞

**现状：** V5 在数据模型中为所有表添加了 `tenant_id` 字段，这是好的。但隔离策略的实现方案仍然模糊。

**具体问题：**

| 问题 | 说明 |
| :--- | :--- |
| **隔离实现方式未明确** | 是用 PostgreSQL Row-Level Security（RLS）还是应用层 ORM 全局 scope？两者的性能和安全性差异很大 |
| **跨租户数据共享场景未定义** | 窗型模板库是否跨租户共享？公共型材数据是否所有租户可见？ |
| **租户数据导出/迁移** | 租户退订后数据如何处理？是否支持数据导出？ |
| **超级管理员的跨租户访问** | 平台运营人员是否需要跨租户查看数据？如何实现？ |

### 3.3 业务流程的"最后一公里"问题

V5 定义了很多模块，但模块之间的**衔接流程**仍有断裂：

| 断裂点 | 说明 | 影响 |
| :--- | :--- | :--- |
| **设计 → 算料 → 报价** 的自动化链路 | V5 分别定义了设计、算料、报价三个模块，但没有明确"用户在画图页面点击保存后，算料是否自动触发？报价是否自动更新？" | 开发团队不知道这三个模块是手动触发还是自动联动 |
| **订单变更 → 生产联动** | 订单变更接口已定义，但变更后如果已经下料了怎么办？已切割的型材如何处理？ | 生产环节的变更回退逻辑缺失 |

| **财务模块与业务模块的对账** | 收款记录和订单金额之间没有自动对账机制 | 财务数据可能与业务数据不一致 |

## 四、文档一致性问题（跨文档冲突）

### 4.1 数据模型冲突

| 冲突点 | PRD_V5 定义 | TEAM-SPEC 定义 | 建议 |
| :--- | :--- | :--- | :--- |
| **主键类型** | `VARCHAR(36)` UUID | `id: string`（前端 nanoid，后端 UUID） | 统一为 `BIGINT AUTO_INCREMENT`（DEVELOPMENT-PLAN 的建议）或统一为 UUID，三份文档必须一致 |
| **设计数据存储** | `design_data_json JSONB` | `design_json JSONB` | 字段名不一致，统一为 `design_data_json` |
| **项目表名** | `order`（订单） | `projects`（项目） | V5 用"订单"作为顶层容器，TEAM-SPEC 用"项目"。这是概念层面的冲突——一个订单包含多樘窗，还是一个项目包含多樘窗？需要明确 |
| **软删除字段** | `is_deleted BOOLEAN` | `deleted_at TIMESTAMPTZ`（DEVELOPMENT-PLAN 建议） | 两种方案各有优劣，必须统一。建议用 `deleted_at`（可记录删除时间） |
| **用户角色** | `admin/designer/worker/installer/finance` | `admin/designer`（仅两种） | TEAM-SPEC 的角色体系过于简化，需要与 V5 对齐 |

### 4.2 API 定义冲突

| 冲突点 | PRD_V5 | TEAM-SPEC |
| :--- | :--- | :--- |
| **基础路径** | `/api/v1/` | `/api/v1/` | 一致 |
| **认证接口** | `POST /api/v1/auth/login` | `POST /auth/login`（缺少 `/api/v1` 前缀） | TEAM-SPEC 的路径不完整 |
| **设计数据接口** | `GET /api/v1/orders/{orderId}/designs` | `POST /projects/:projectId/designs` | 父资源不同（order vs project），需要统一 |
| **响应格式** | `{ "code": 200, "message": "success", "data": {...} }` | `{ "data": {...}, "message": "ok" }` | 响应结构不一致，V5 有 `code` 字段，TEAM-SPEC 没有 |

### 4.3 术语不一致

## 五、产品层面的战略性缺失

以下不是"文档写漏了"的问题，而是产品规划层面的战略性缺失。作为软件负责人，这些问题会影响产品的市场竞争力和商业成功。

### 5.1 缺少竞品分析

V5 没有任何竞品分析章节。门窗行业的 SaaS 软件已有多个成熟竞品：

| 竞品 | 定位 | 核心优势 | 我们的差异化 |
| :--- | :--- | :--- | :--- |
| **画门窗 Pro** | 行业领先的门窗设计算料软件 | 算料公式成熟、型材库完善、用户基数大 | **需要定义** |
| **门窗大师** | 门窗 ERP + 设计一体化 | 全流程管理、生产排程 | **需要定义** |
| **酷家乐（门窗版）** | 3D 家装设计平台 | AI 渲染效果好、品牌知名度高 | **需要定义** |
| **云图门窗** | 轻量级门窗设计工具 | 上手简单、价格低 | **需要定义** |

**建议：** 补充竞品分析章节，明确我们的差异化定位。如果没有清晰的差异化，产品很难在已有竞品中突围。

### 5.2 缺少用户画像和使用场景

V5 定义了 5 种用户角色（管理员、设计师、车间工人、安装工、财务），但没有定义：

| 缺失内容 | 说明 |
| :--- | :--- |
| **用户画像** | 目标用户是什么规模的门窗厂？年产值多少？几个人？IT 水平如何？ |
| **核心使用场景** | 用户一天的工作流程是什么？在哪些环节使用我们的软件？ |
| **付费意愿分析** | 门窗厂愿意为软件付多少钱？现在用什么工具？痛点是什么？ |
| **用户增长路径** | 从免费版到付费版的转化路径是什么？什么功能是"钩子"？ |

### 5.3 缺少北极星指标

产品没有定义核心衡量指标。建议定义：

| 指标 | 定义 | 目标值 |
| :--- | :--- | :--- |
| **WAU（周活跃用户）** | 每周至少使用一次设计功能的用户数 | V1.0 上线 3 个月后 ≥ 500 |
| **设计完成率** | 创建设计后完成保存的比例 | ≥ 80% |
| **算料准确率** | 算料结果与实际用料的偏差 | ≤ 3% |
| **付费转化率** | 免费用户转为付费用户的比例 | ≥ 5% |
| **客户留存率** | 月度付费客户续费率 | ≥ 90% |

### 5.4 缺少 MVP 定义

V5 定义了 14 个模块，但没有明确**最小可行产品（MVP）**包含哪些模块。如果所有模块都要做完才上线，周期太长（32 周）。

**建议的 MVP 范围：**

| 包含 | 不包含（V1.1+） |
| :--- | :--- |
| 用户认证 + 多租户 | 车间管理 |
| 画图设计 + 2D/3D | 仓库管理 |
| 算料引擎 | 采购管理 |
| 报价单 | AI 渲染 |
| 订单管理 | 阳光房 3D |
| 客户管理 | AR 拍照 |
| 导出 PDF | 设备对接 |
| — | 数据报表 |

MVP 目标：**让门窗厂能用这个系统完成"设计→算料→报价→下单"的核心闭环**，预计 12-16 周可交付。

## 六、前端画图模块的深度问题

作为产品的核心差异化模块，画图渲染部分需要特别关注。

### 6.1 当前实现 vs V5 要求的差距

| 维度 | 当前实现（ARCHITECTURE.md） | V5 要求（7.10 节） | 差距 |
| :--- | :--- | :--- | :--- |
| **渲染引擎** | 手写 SVG | 推荐 Konva.js | 需要评估是否迁移，SVG 在复杂场景下性能不如 Canvas |
| **交互模型** | 仅支持模板驱动 + 基础选择 | 三位一体（模板驱动 + 自由拖拽 + 精确输入） | 自由拖拽和精确输入尚未实现 |
| **状态管理** | React Context + useReducer | 推荐 Zustand | 当前方案在复杂场景下可能有性能问题 |
| **移动端** | 未适配 | 要求 44x44pt 触摸目标 + 手势支持 | 完全缺失 |
| **分层渲染** | 单层 SVG | 要求多层 Canvas（静态层/动态层/辅助线层/标注层） | 架构差距大 |
| **离屏渲染** | 无 | 要求预渲染常用图形为精灵 | 未实现 |
| **Web Worker** | 无 | 要求算料等计算放入 Worker | 未实现 |

### 6.2 关键技术决策待定

| 决策 | 选项 | 建议 |
| :--- | :--- | :--- |
| **是否从 SVG 迁移到 Canvas** | (A) 保持 SVG，优化性能 (B) 迁移到 Konva.js (C) 迁移到 Fabric.js | 建议做 PoC 对比，重点测试 100+ 组件场景下的渲染帧率 |
| **是否从 useReducer 迁移到 Zustand** | (A) 保持 useReducer (B) 迁移到 Zustand | 建议迁移，Zustand 的选择性订阅可以减少不必要的重渲染 |
| **3D 预览是否需要升级** | (A) 保持 Three.js 基础渲染 (B) 引入 PBR 材质和环境贴图 | V1.0 保持现状，V2.0 再升级 |

## 七、文档体系的结构性问题

### 7.1 文档之间的关系不清晰

当前有 5 份文档，但它们之间的关系和优先级没有明确定义：


## 九、总结

### 9.1 文档体系的优势

1. **覆盖面广**：14 个功能模块基本覆盖了门窗厂全流程，从设计到生产到售后
2. **数据模型较完整**：41 张数据表的定义为后端开发提供了良好的起点
3. **API 定义数量充足**：23 组 API 覆盖了主要业务场景
4. **非功能性需求意识强**：性能、安全、监控、国际化都有考虑
5. **V5 对 V3 缺失的补齐力度大**：50% 完全补齐，30% 部分补齐

### 9.2 文档体系的核心短板

1. **核心算法模块（算料、下料优化）缺乏可执行的规格定义**——这是产品的灵魂，不能含糊
2. **多份文档之间存在冲突**——数据模型、API 定义、术语在不同文档中不一致
3. **缺少产品战略层面的文档**——竞品分析、用户画像、北极星指标、MVP 定义
4. **前端画图模块的技术选型未确定**——SVG vs Canvas 的决策影响整个架构
5. **模块间的衔接流程有断裂**——设计→算料→报价→生产的自动化链路未定义

### 9.3 一句话结论

> **V5 文档体系的广度已经足够启动开发，但深度不够——核心差异化模块（算料引擎、下料优化、画布引擎）需要再投入 1-2 周做规格定义和技术 PoC，才能真正"交给团队开干"。**
